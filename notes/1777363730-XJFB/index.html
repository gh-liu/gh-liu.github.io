Go: defer

2026-04-282026-04-283 出链
CSGO

defer 用来登记一个在当前函数退出前执行的调用。

它通常和 panic / recover 一起理解：defer 决定了清理逻辑在正常返回和异常展开时都能被执行。

defer 是什么

语义上不是现在执行，而是：

  1. 先计算调用表达式需要的值
  2. 把这次调用登记到当前函数的 defer 栈里
  3. 等当前函数返回时再执行

所以：

  • defer 绑定的是当前函数
  • 不是绑定当前代码块
  • 不是绑定调用者

什么时候执行

当前函数退出时执行，包括两种退出路径：

  1. 正常 return
  2. 因 panic 开始栈展开

并且多个 defer 按 LIFO 执行：后注册，先执行。

参数求值时机

这是 defer 最容易搞错的一点：

  • defer 的调用延后执行
  • 但实参求值发生在 defer 语句出现的那一刻

defer 和 return 的关系

return 不是立刻离开函数，而是大致经历：

  1. 先计算返回值
  2. 再执行 defer
  3. 最后真正返回给调用者

这也是为什么 defer 能修改命名返回值。

defer 和 panic/recover

panic 发生后，runtime 会沿当前 goroutine 的调用栈展开，并逐层执行 defer。

recover 之所以能工作，就是因为它运行在这些 defer 里。

所以：

  • defer 是 panic 时的清理钩子
  • defer 也是 recover 的承载位置

常见用途

  1. 资源释放：defer file.Close()
  2. 解锁：defer mu.Unlock()
  3. 结束 tracing / profiling：defer task.End()
  4. 统计耗时 / 记录日志
  5. 在 goroutine / handler 边界 recover

常见坑

循环里 defer

defer 绑定函数，不是绑定循环迭代。

这不会在每轮循环结束就关闭文件，而是要等整个函数返回才统一关闭。

如果循环很多次，可能拖太久甚至耗尽 fd。

更稳妥的方式：

  • 显式 Close()
  • 或把单次迭代提到一个小函数里

闭包捕获变量

这里要非常清楚你是想要：

  • defer 时立即冻结参数
  • 还是执行 defer 时再读变量

闭包捕获的是变量，不是变量当时的值。

误以为 defer 很零成本

现代 Go 对 defer 做了很多优化，但它仍然不是免费。

经验上：

  • 绝大多数业务代码优先写清楚
  • 极热路径里再考虑是否要手工展开

runtime 视角

从 runtime 结构看，goroutine (g) 上挂着 _defer 链。

也就是说，defer 不是语法糖到源码级展开完就结束；runtime 需要知道当前 goroutine 挂了哪些延迟调用，才能在：

  • 正常返回时执行
  • panic 展开时执行
  • traceback / 栈复制等场景中维护一致性

最佳实践

  1. Open/Lock/Start 之后，尽快写下对应的 defer Close/Unlock/Stop
  2. 先判断错误，再 defer 清理，不要对 nil 资源 defer 方法调用
  3. 需要修改返回值时，优先显式说明，避免把 defer 写得太隐晦
  4. 循环里的 defer 要小心资源生命周期
  5. 热路径先测再优化，不要凭感觉去掉 defer
