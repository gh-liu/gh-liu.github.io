grpc

gRPC = HTTP/2 + Protobuf + IDL/codegen 的工程化 RPC 方案，目标是在跨进程/跨机器调用里，同时拿到类型约束、性能和流式能力。

S 微服务内部经常需要低延迟、跨语言地调用远程接口。 C 远程调用想像本地函数一样清晰，但必须处理网络、性能、类型和协议演进问题。 Q 如何让服务间调用同时具备强类型、跨语言、高性能和流式能力？ A gRPC 用 HTTP/2 + Protobuf + .proto codegen，把远程调用组织成工程化 RPC。

核心组成

  • RPC：让远程调用尽量接近本地函数调用
  • HTTP/2：提供多路复用、二进制分帧、Header 压缩、长连接
  • Protobuf：紧凑的二进制编码 + 明确的 schema
  • IDL：用 .proto 统一定义消息结构和服务接口
  • Codegen：自动生成客户端和服务端骨架代码

最小例子

调用流程大致是：

  1. IDL：定义 .proto
  2. Codegen：生成代码
  3. 服务端实现接口
  4. 客户端像调用本地方法一样发起 RPC
  5. 底层由 gRPC 负责序列化、传输、超时、状态码等细节

生成代码通常包含三类东西：

  • client stub：给客户端调用，负责发请求和收响应
  • server skeleton/interface：给服务端实现，并注册到 gRPC server
  • message types：请求和响应的数据结构，两端共用

四种调用模式

  1. Unary RPC：请求一次，返回一次；最像普通函数调用
  2. Server Streaming：客户端发一个请求，服务端持续返回多条消息
  3. Client Streaming：客户端持续发多条消息，服务端最后汇总返回一次
  4. Bidirectional Streaming：双方都可以持续发送消息，互不阻塞

这也是 gRPC 相比很多 REST 风格接口更强的地方：它天然适合流式通信。

场景

适合：

  • 微服务内部调用
  • 多语言后端协作
  • 对性能、带宽、延迟较敏感的系统
  • 需要稳定 schema 和代码生成的团队
  • 需要 streaming 的场景

不适合：

  • 面向第三方开放的公共 API：很多场景 REST/JSON 更通用，调试门槛更低
  • 浏览器直接调用：浏览器对原生 gRPC 支持有限，通常需要 gRPC-Web
  • 只需要很简单 CRUD 接口的系统：引入 proto、codegen、网关可能反而增加复杂度

和 REST/HTTP API 的关系

不是谁替代谁，而是边界不同：

  • 外部 API：常用 REST/JSON，兼容性和可调试性更重要
  • 内部服务通信：常用 gRPC，性能、schema、生成代码更重要

很多系统最终是：

  • 对外暴露 REST/HTTP API
  • 内部服务之间使用 gRPC

为什么 gRPC 常常更快

  • Protobuf 是二进制格式，比 JSON 更紧凑
  • HTTP/2 支持单连接多路复用，减少连接成本
  • 代码生成减少了手写编解码和胶水代码
  • 强 schema 让接口更稳定，也更适合机器处理

所以它常见于服务间内部通信，而不是给人直接调试的开放 API。

需要记住的代价

  • 远程调用不是本地调用：要显式处理超时、取消、重试、幂等性
  • schema 演进要守兼容规则：字段编号不能乱改
  • 调试比 JSON 接口更黑盒，需要专门工具
  • HTTP/2 和连接管理也会带来运维复杂度
