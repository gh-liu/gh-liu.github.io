S 什么场景 微服务、RPC 调用、消息队列、数据库访问混在一起时，一次用户请求通常会经过很多组件。只看单机日志，往往知道某处报错了，但不知道这次请求到底走了哪条路径、卡在哪一跳。 C 什么冲突 系统拆得越细，排障越难。日志擅长看单点细节，指标擅长看整体趋势，但它们都不天然回答这一个请求经历了什么。所以需要 trace 把一次请求在全链路上的行为串起来。 Q 什么问题 trace 是什么？它和 log、metric 有什么区别？ A 回答问题 trace 可以理解成一次请求的完整调用链记录。

  • Trace：对应一次完整请求，比如用户点一次下单。
  • Span：对应链路里的一个操作片段，比如 API 网关、订单服务、库存服务、MySQL 查询，各自都可以是一个 span。
  • Trace ID：整条链路共享的唯一标识，用来把不同服务里的 span 串起来。
  • Span ID：单个 span 的标识，用来描述父子调用关系。

比如一次下单请求可能是：

Gateway -> Order Service -> Inventory Service -> Payment Service -> MySQL

如果支付慢了，trace 能直接看到慢的是 Payment Service 这个 span；如果库存服务调用失败，也能看到错误是在它的子 span 上发生，而不是只看到某台机器打了一条报错日志。

所以三者的区别可以粗看成：

  • log：看某个点发生了什么。
  • metric：看一段时间内整体是否异常。
  • trace：看某一次请求完整经过了什么路径、每一段花了多久。

可观测性里的 trace，核心价值就是把离散的事件还原成完整的请求路径，方便定位慢请求、失败请求和跨服务问题。

为什么有日志还需要 trace

  • 日志能告诉你这里发生过什么，但前提是你知道该去哪个服务、哪台机器、哪个时间点查。
  • 当一次请求会穿过很多服务时，相关日志会散落在不同地方，人工拼接很慢。
  • trace 先把一次请求串成一条链，你再回头看具体 span 的日志，会更快定位问题。

可以简单记成：

  • log 解决局部细节
  • trace 解决全链路关联

OpenTelemetry 里怎么传递

https://opentelemetry.io/docs/concepts/signals/traces

工程里通常会给每次请求带上一份 trace context，里面常见有：

  • trace_id
  • span_id
  • parent_span_id

当服务 A 调服务 B 时，会把这份上下文一起传过去。这样 B 新建 span 时，就知道自己的父 span 是谁，最后平台才能把所有 span 拼成一棵调用树。

常见过程可以记成：

  1. 请求进入网关，创建根 span。
  2. 网关调用订单服务，把 trace context 透传过去。
  3. 订单服务创建自己的子 span，再继续传给库存、支付、数据库。
  4. 所有 span 上报后，就能看到一条完整 trace。

所以 trace 本质上不是多打一种日志，而是给整条调用链建立统一上下文。
