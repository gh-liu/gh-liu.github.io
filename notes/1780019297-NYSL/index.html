管理k8s应用

  • S：K8s 应用通常包含多种资源（多份 YAML），不同环境有差异（资源结构相同，但有一些配置不同）
  • C：手写和复制配置易漂移，升级回滚难追踪（涉及多份资源变更，缺少整体概念），无法复用
  • Q：如何把应用交付变成一个交付单元，安装、升级、回滚（可模板化、参数化、版本化、复用...）
  • A：Helm 用 Chart 模板化资源（k8s应用），用 Values 注入环境配置，并以 Release 记录安装、升级、回滚

把 Kubernetes YAML 资源集合，抽象成可参数化、可复用、可版本化的应用包，并通过 release 管理应用的安装、升级与回滚。

核心抽象

  • Chart：应用包，组合一组 K8s 资源模板、默认 Values 和元数据。
  • Values：参数输入，解决环境差异和配置复用问题。
  • Release：Chart 的一次安装实例，解决升级、回滚、状态追踪问题。
  • Repository：Chart 分发仓库，解决应用包发现和复用问题。
  • Template：K8s YAML 模板，解决重复手写资源清单的问题。

Helm 可以理解为 K8s 之上的应用交付抽象层：K8s 管资源，Helm 管由一组资源组成的应用包及其生命周期。

心智模型

Helm 以 Chart 为交付单位，以 Release 为生命周期管理对象。

helm cli 常用命令

心智模型，按helm的核心对象来理解： Repository -> Chart -> Values -> Rendered YAML -> Release

helm install 会把 Chart + Values 渲染成 K8s 资源并提交给 API Server；Helm 管 Release 生命周期，K8s 负责实际运行。
