一致性哈希

2026-04-272026-04-27 17:04:57
CS

https://en.wikipedia.org/wiki/Consistent_hashing

S 分布式缓存、分库分片、CDN 节点选择这类场景里，数据或请求需要稳定地映射到多台机器 C 如果直接用 hash(key) % N，一旦节点数 N 变化，大量 key 都会重新映射，扩容缩容时 rebalance 成本很高 Q 有没有一种分配方式，节点增减时不要全量迁移？ A 把 key 和节点都映射到同一个哈希环上，key 顺时针找到的第一个节点就是归属节点；这样节点变化通常只影响相邻的一小段区间

核心思想

  • 把哈希空间首尾相连看成一个环
  • 机器节点先映射到环上
  • 数据 key 也映射到环上，顺时针遇到的第一个节点负责存储

为什么工程上常用

  • 扩容：新节点加入后，通常只接管它顺时针前驱到自己之间那段数据
  • 缩容：节点下线后，它负责的数据顺时针交给后继节点
  • 迁移范围小：相比普通取模，一致性哈希不需要大规模重算全部 key 的归属

所以它常用在需要在线扩缩容的系统里，本质是在降低 rebalance 代价。

工程注意点

  • 数据倾斜：节点直接上环可能分布不均，常见做法是给每台机器放多个虚拟节点，把负载打得更匀
  • 热点 key：一致性哈希只能解决“分布稳定”和“迁移变少”，不能天然消除热门 key 打爆单机的问题；热点通常还要配合副本、局部缓存或热点拆分

适用场景

  • 分布式缓存路由
  • 分库分片
  • 对象存储 / CDN / 代理节点选择
