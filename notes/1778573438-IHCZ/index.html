Types and Programming Languages (TAPL) — Benjamin C. Pierce https://www.cis.upenn.edu/~bcpierce/tapl/

https://zackoverflow.dev/writing/tapl

  • 是什么（What）— 类型系统是一种轻量级形式化方法，通过在编译期对程序中各项的求值结果进行分类，自动证明某类运行时错误不可能发生
  • 为什么（Why）— 在表达力（能写多少程序）与可判定性（能静态证明多少性质）之间做权衡；用类型把"不该发生的状态变得不可表达（make illegal states unrepresentable）"
  • 怎么做（How）— 从无类型 λ 演算出发，逐步加入简单类型、子类型、递归类型、多态、存在类型、高阶类型，每一步都给出语法、求值规则、定型规则，并证明 Progress + Preservation = Soundness

核心观点：

1. 类型系统的定义（Pierce 的经典表述）

A type system is a tractable syntactic method for proving the absence of certain program behaviors by classifying phrases according to the kinds of values they compute.

四个关键词：

  • tractable（可处理）：定型算法必须高效、可判定
  • syntactic（语法的）：基于程序文本的形状，而非运行时行为
  • proving absence（证明不发生）：保证某类错误不出现，但不保证程序正确
  • classifying（分类）：把表达式按"它会算出什么"归类

2. 形式化方法的三件套

TAPL 中所有的语言定义都遵循同一套范式：

  • 语法（Syntax）：用 BNF 描述项（terms）和类型（types）
  • 求值规则（Evaluation）：小步语义 t → t'，定义程序如何一步步执行
  • 定型规则（Typing）：判断式 Γ ⊢ t : T，在上下文 Γ 下项 t 具有类型 T

3. 类型安全 = Progress + Preservation

这是全书最核心的定理模板：

  • Progress（进展性）：良类型的项要么是值，要么能继续求值（不会卡住）
    • If ⊢ t : T, then either t is a value or ∃ t'. t → t'
  • Preservation（保持性，又叫 Subject Reduction）：求值一步后类型不变
    • If ⊢ t : T and t → t', then ⊢ t' : T
  • 二者合起来：良类型程序永远不会进入"卡住状态（stuck state）"——即非值且无法继续求值的状态

4. λ 演算是一切的基石

  • 无类型 λ 演算（untyped λ-calculus）：图灵完备，但 Ω = (λx. x x)(λx. x x) 这类项会发散
  • 简单类型 λ 演算（STLC）：加上 → 类型后，所有良类型项强归约（strongly normalizing），但失去图灵完备性
  • 这是表达力 vs 安全性的根本权衡

5. 类型系统的演化阶梯

TAPL 按"复杂度递增"组织全书，每一章都是给前一个系统加一个特性：

  • λ→：简单类型
  • λ<:：加子类型（subtyping）
  • λµ：加递归类型（recursive types）
  • λ2/System F：加全称多态（universal polymorphism / parametric polymorphism）
  • λ∃：加存在类型（existential types）→ 抽象数据类型
  • Fω：加高阶多态（type operators）
  • Fω<:：把所有特性合在一起，是现代语言（Scala、Rust trait、TypeScript）的理论原型

6. Curry-Howard 同构

  • 命题即类型，证明即程序（Propositions as Types, Proofs as Programs）
  • 类型 A → B 对应蕴含 A ⇒ B
  • 类型 A × B 对应合取 A ∧ B
  • 类型 A + B 对应析取 A ∨ B
  • 多态对应全称量化 ∀
  • 存在类型对应存在量化 ∃
  • 这是类型论与逻辑学的深层联系，也是 Coq、Agda、Lean 等定理证明器的基础

启发点（关键洞察）：

  1. 类型不是"标注"，而是"证明"：每一条定型规则都是一条推理规则；写出 e : T 等价于在某种逻辑里给出了一个证明。把类型系统看成"嵌在编译器里的小型证明助手"。

  2. "卡住"才是真正的错误：TAPL 把"运行时错误"重新定义为"求值卡住"。1+true 之所以是错的，不是因为它"语义错误"，而是因为没有任何求值规则能继续推进。这种统一让"安全"成了一个可证明的数学性质。

  3. 进展性 + 保持性是万能模板：无论你设计什么新类型系统，都按这两条定理来证明它的安全性。如果证不出来，说明系统有漏洞——通常是某条定型规则太松或某条求值规则没覆盖。

  4. 子类型即安全替换（Liskov 的形式化）：S <: T 意味着任何需要 T 的地方都可以用 S。函数类型的子类型是逆变-协变的：(T1 → T2) <: (S1 → S2) 当且仅当 S1 <: T1 且 T2 <: S2。这是 OOP 中"参数逆变、返回值协变"规则的来源。

  5. 递归类型需要"折叠/展开"：µX. T 类型必须显式 fold/unfold（等价递归 vs 同构递归）。Rust 的 Box<T> 解决无限大小问题，本质就是引入一层显式不动点。

  6. 多态 = 类型抽象 + 类型应用：System F 的 ΛX. t 和 t [T] 与 λ 演算的 λx. t 和 t e 完全对偶。类型推断在 System F 中是不可判定的，所以实际语言（ML、Haskell）用 Hindley-Milner 限制为前束（rank-1）多态。

  7. 存在类型 = 抽象数据类型（ADT）：{∃X, {a: X, f: X → Int}} 表示"存在某个类型 X，对外暴露字段 a 和函数 f"。这正是模块、接口、dyn Trait 的形式化基础。

  8. 类型系统是设计语言的"骨架"：你想让语言支持什么特性（继承、泛型、模式匹配、effect），就先在类型系统里把它表达清楚，再考虑语法糖。Rust 的所有权、TypeScript 的结构类型、Scala 的隐式参数，都是类型系统层面的设计决策。

  9. 保守即安全：类型系统拒绝的程序中，绝大部分其实在运行时是安全的（如 if true then 1 else "abc"）。这是必然的代价：类型检查是不完备的（rejects some safe programs），但必须是可靠的（never accepts an unsafe program）。

  10. 形式化不是奢侈品：写出语法、求值规则、定型规则、证明 Progress + Preservation——这套训练能让你在设计 DSL、API、配置语言时，瞬间识别"这里语义不闭合""那条规则会破坏类型安全"。

行动：

  1. 读 TAPL 的方式：

    • 第 1-2 部分（Ch 1-7）必读：建立无类型 λ 演算与简单类型系统
    • 第 3 部分（Ch 8-12）必读：STLC、类型安全证明、扩展（pair、sum、recursion）
    • 第 4 部分（Ch 13-18）按需：引用、异常、子类型、递归类型
    • 第 5 部分（Ch 22-23）：Hindley-Milner 类型推断 + System F
    • 第 6 部分（Ch 28-32）：高阶多态 + 子类型，挑战大
    • 配合实现：每章末尾的 OCaml 实现一定要跑、改、扩展

  2. 做配套练习：

    • 用 OCaml/Haskell/Rust 实现 STLC 解释器
    • 实现类型检查器，至少写到 System F 子集
    • 加一个新特性（如 let rec、record、sum type）并证明 Progress/Preservation

  3. 画推导树：每条定型规则、每个例子，手动画出推导树，理解"类型是怎么被推出来的"

  4. 关联实际语言：

    • Rust 借用检查 ↔ 仿射类型（affine types，TAPL 未深入但可延伸）
    • TypeScript ↔ 结构子类型 + 联合类型 + 条件类型
    • Haskell ↔ Hindley-Milner + 类型类
    • Scala ↔ Fω<: 的近似

  5. 学习路径延伸：

    • ATTAPL（Advanced Topics in Types and Programming Languages）：TAPL 续作，涵盖依赖类型、模块、副作用类型
    • Software Foundations（Pierce 团队）：用 Coq 把 TAPL 的所有证明形式化重做一遍
    • PLFA（Programming Language Foundations in Agda）：Agda 版本

  6. 写自己的语言：基于 STLC 设计一个小型 DSL（如配置语言、查询语言），把"类型设计先行"作为习惯

金句：

  1. "A type system is a tractable syntactic method for proving the absence of certain program behaviors." — Pierce 给类型系统的定义，每一个词都不可省

  2. "Well-typed programs cannot go wrong." — Robin Milner，类型安全的口号式表述

  3. "Make illegal states unrepresentable." — Yaron Minsky，类型系统在工程上的最高指导原则

  4. "Types are the leaven of computer programming; they make it digestible." — Pierce，类型是编程语言的"酵母"

  5. "Progress + Preservation = Type Soundness" — 一切类型安全证明的万能公式

  6. "Propositions as Types, Proofs as Programs." — Curry-Howard 同构，连接逻辑与计算的桥梁

  7. "The design of a type system is the design of a language." — 类型系统不是附加物，而是语言设计的核心
