0.0.1 · Alloc

https://draveness.me/golang/docs/part3-runtime/ch07-memory/golang-memory-allocator/ http://t.zoukankan.com/zpcoding-p-13259943.html

  1. 类似TCMalloc
  2. 小分配（32KB以内），会四舍五入到对应的多种不同大小的内存块（1B~32KB）之一，内存页面会分割成对应某种大小的对象，使用bitmap管理；
  3. 分配器的数据结构有: fixalloc, mheap(分配堆，管理页（8KB）), mspan(一连串使用中的页), mcentral(指定大小的mspan), mcache(每个p中的可用的mspan), mstats（分配统计）

分配小对象:

  1. 四舍五入成一个等级大小的内存，去P中的mcache中找到一个可用的mspan，在mspan的位图中找到一个可用的槽，分配其，此操作不用锁；
  2. mspan没有可用的槽，从mcentral中获取一个新的mspan，需要锁；
  3. mcentral是空的，则从mheap中获取一连串的页
  4. 如果mheap是空的或者没有足够大小的页，从操作系统获取页（至少1MB），需要调用系统调用

清除mspan，释放对象：

  1. 因为分配而清除的mspan, 为了满足分配而返回给mcache
  2. 否者，如果mspan依旧有分配的对象在，则放在mcentral中对应的mspan大小列表中
  3. 否者，mspan中的对象被释放，mspan被返回给mheap

分配/释放大对象避开mcache和mcentral，直接与mheap交互

mspan.needzero? 零值?

虚拟内存布局： heap由一系列的arenas组成，在64位上为64MB，在32位上为4MB; 每一个arenas关连一个heapArena对象，其存储了arena的元数据: heap位图对应arena中的所有word，span图对应arena中的所有页，heapArea不在堆上分配； 由于arenas是对其的，因此可以把内存空间看成一连串的arenas，mheap._areans映射着一系列的heapArena对象；这个映射由二维数组组成，L1和许多L2，许多机器上可能只有单个L2映射；

mksizeclasses.go 生成小分配大小的等级，每个级别之间差12.5%，

以32KB为界限，分为大小对象； tiny ：size < 16 bytes && has no pointer(noscan) small ：has pointer(scan) || (size >= 16 bytes && size <= 32 KB) large ：size > 32 KB 其中小对象如果无指针且小于16B，视作微对象； 微对象：将几个微小对象分配请求整合到一个内存块内； 小对象：根据大小，找到对应的sizeclass，再获取到size，再获取到span的class，再从mcache中拿出一个span 大对象：直接从mheap分配span

使用span机制来减少碎片,每个span至少为一个页,一共有70个size范围, 8byte-32KB, 每个size有两种类型(scan和noscan, 表示分配的对象是否会包含指针) 多层次Cache来减少分配的冲突: per-P无锁的mcache, 全局672个对应不同size的span的后备mcentral, 全局1个的mheap. mheap中以treap的结构维护空闲连续page. 归还内存到mheap时, 连续地址会进行合并. (1.11之前采用类似伙伴系统维护<1MB的连续page, treap维护>1MB的连续page) go代码分配内存优先从当前p的mcache对应size的span中获取; 有的话, 再从对应size的mcentral中获取一个span; 还没有的话, 从mheap中sweep一个span; sweep不出来, 则从mheap中空闲块找到对应span大小的内存. mheap中如果还没有, 则从系统申请内存. 从无锁到全局1/(672)粒度的锁, 再到全局锁, 再到系统调用. stack分配也是多层次和多class的.减少分配的锁争抢, 减少栈浪费. 对象由GC进行释放. sysmon会定时把mheap空余的内存归还给操作系统

在 GC 的过程中同时运行的 g 称为mutator，mutator assist机制就是 g 辅助 GC 做一部分工作的机制。 辅助 GC 做的工作有两种类型，一种是标记（Mark），另一种是清除（Sweep）。

使用写屏障的并发标记与清除； 非分代、非压缩；

分配内存是通过每个p中大小隔离开的分配区域完成的，在消除常见情况下的锁的同时，最小化内存碎片。

  1. GC 执行清除终止 a. STW，所有p到GC安全点 b. 清除任何未清除过的spans，只有在预期时间之前，强制执行此次GC时，才会有未清除的span
  2. GC 执行标记阶段 a. 准备标记阶段，将gcphase设置为_GCmark（从_GCoff开始），启用写屏障，启用 mutator assist，并对根标记作业进行排队。在所有P都启用写屏障之前，不会扫描任何对象，扫描是通过STW完成的？？ b. Start the world，从现在开始，GC 工作由调度器启动的标记worker和作为allocation的一部分执行的assists来完成；写屏障着色被覆写的指针和任何新指针值；新分配的对象立即被标记为黑色 c. GC 执行根标记作业。包括：扫描所有栈，着色所有全局变量，以及着色堆外运行时数据结构中的任何堆指针；扫描栈会停止goroutine，对goroutine栈中找到的任何指针进行着色，然后恢复goroutine d. GC 耗尽灰色对象的工作队列，将每个灰色对象扫描为黑色，并对在该对象中找到的所有指针进行着色（反过来可能会将这些指针添加到灰色工作队列中） e. 由于 GC 工作分散在本地缓存中，因此 GC 使用分布式终止算法来检测何时不再有根标记作业或灰色对象（参见gcMarkDone函数）。此时，GC 状态转换到标记终止（gcMarkTermination）
  3. GC 执行标记终止 a. STW b. 将gcphase设置为_GCmarktermination，并禁用 workers 和 assists。 c. 进行内务整理，如flushing mcaches
  4. GC 执行清除阶段 a. 准备清除阶段，将gcphase设置为_GCoff，设置清除状态并禁用写屏障 b. Start the world，从现在开始，新分配的对象是白色的，如有必要，在使用spans前分配清除spans c. GC 在后台进行并发清除并响应内存分配
  5. 当充足的内存分配发生时，重复上面以 1 开始的步骤，参见关于 GC rate 的讨论。

并发清除： 在后台goroutine中，堆被惰性（当 goroutine 需要另一个span时）且并发地逐个span扫描（这有助于不是CPU密集型的程序）。在STW 标记终止的结尾，所有的span都被标记为需要清除。

后台清除 goroutine 简单地逐个清除span。

为了避免在存在未清除的span时请求更多的OS内存，当goroutine需要另一个span时，它首先尝试通过清除来回收这些内存。 当 goroutine 需要分配一个新的小对象span时，它会清除相同大小的小对象span，直到释放至少一个对象为止。 当 goroutine 需要从堆中分配大对象span时，它会清除span，直到将至少那么多页面释放到堆中。 有一种情况，这可能是不够的：如果 goroutine 清除并释放两个不相邻的单页span到堆中，那么它将分配一个新的双页span，但是仍然可以有其他单页未清除的span，可以组合成双页的span。

确保在未清除的span上不进行任何操作（这会破坏 GC 位图中的标记位）至关重要。 在 GC 期间，所有mcache都被刷新到中央缓存中，因此它们是空的。 当一个 goroutine 抓取一个新的span到mcache时，goroutine会清除mcache。 当 goroutine 显式释放对象或设置finalizer时，goroutine 确保span已经清除（通过清除或者等待并发清除完成）。 finalizer goroutine仅在所有span已经清除时才开始。 当下一次 GC 启动时，它将清除所有尚未清除的span（如果有的话）。

下一次 GC 是在我们分配了与已经使用的内存成正比的额外内存时触发； 该比例由GOGC环境变量控制（默认为100）。 如果GOGC=100，而我们当前使用的内存的是4M，那么当达到8M时，触发GC。

为了防止在扫描大型对象时出现长时间的暂停，与提高并行性，垃圾收集器将大于maxObletBytes的对象的扫描作业分解为最多maxObletBytes的oblets数组。 当扫描遇到大对象时，它只扫描第一个oblet，并将其余oblets作为新的扫描作业排队。

tri-color mark and sweep.

  1. Mark Setup - STW 写屏障，所有g进入安全点
  2. Marking - Concurrent 占据1/4可用的cpu GC独占一个GMP 找到堆内存的根对象 通过根对象遍历堆内存图 P1 in GC,other ps still work 可使用g协助标记
  3. Mark Termination - STW 关闭写屏障，清楚任务 计算下次GC目标
  4. Sweeping 当用户g尝试再对上分配新值时发生
