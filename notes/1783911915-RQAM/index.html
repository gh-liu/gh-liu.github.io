关系型数据库 PostgreSQL

  • S 业务系统需要保存、查询和修改结构化数据，也需要复杂查询和丰富的数据类型
  • C 状态管理会遇到并发、故障、数据增长、一致性约束和多样化查询
  • Q 如何让业务数据可查询、可修改、可约束、可恢复、可并发，并支持持续扩展？
  • A 用关系表 + SQL + 索引 + MVCC + WAL + VACUUM + 复制

心智模型

PostgreSQL 是一个多进程的关系型数据库：表保存多个行版本，索引指向表中的元组；MVCC 用快照决定版本可见性，WAL 保证崩溃恢复，VACUUM 回收不再可见的旧版本。

与 MySQL/InnoDB 最值得区分的地方：

  • PostgreSQL 的表是 heap，主键也只是一个唯一索引，没有 InnoDB 式聚簇主键
  • 更新通常不是原地覆盖，而是写入一个新元组版本；旧版本留在表内，之后由 VACUUM 回收
  • PostgreSQL 使用服务端多进程模型：通常一个客户端连接对应一个后端进程
  • PostgreSQL 只有一个存储引擎，但通过扩展、丰富的数据类型和多种索引访问方法增强能力

SQL 怎么执行

客户端 → 后端进程 → 解析/重写 → 优化 → 执行 → 表与索引

1. 多进程架构

  • postmaster（主服务进程）监听连接，并为每个客户端派生 backend process
  • 每个后端进程负责该连接的 SQL 解析、规划和执行
  • 进程通过共享内存协作，其中最重要的是 shared_buffers、WAL buffers 和锁表
  • 生产环境通常使用 PgBouncer 等连接池，避免大量短连接频繁创建进程和占用内存

主要后台进程：

进程 作用
checkpointer 定期触发检查点，将检查点之前的脏页落盘
background writer 平滑地把部分脏页写回磁盘，减少后端进程写盘压力
WAL writer 把 WAL buffers 中的记录写入 WAL 文件
autovacuum launcher/worker 自动执行 VACUUM 和 ANALYZE
archiver 开启归档后，将完成的 WAL 段复制到归档位置

2. SQL 执行流水线

  1. Parser：词法、语法分析，生成解析树
  2. Analyzer/Rewriter：解析对象名称和类型，并应用规则、视图等重写
  3. Planner/Optimizer：枚举扫描与连接路径，按成本选择执行计划
  4. Executor：按计划节点拉取和处理元组

PostgreSQL 优化器基于成本估算，不只看有没有索引：小表、低选择性条件或回表成本高时，顺序扫描可能更便宜。

数据怎么组织

Database → Schema → Relation → Fork → Page → Tuple

1. 逻辑与物理组织

  • 一个 PostgreSQL 实例（cluster）可包含多个 database；database 内用 schema 隔离命名空间
  • 表和索引统称 relation，每个 relation 对应若干物理文件
  • relation 除主数据文件外还有不同 fork：
    • main：表或索引数据
    • fsm：Free Space Map，记录页面剩余空间
    • vm：Visibility Map，记录页面是否全部可见、是否已冻结
    • init：unlogged relation 初始化副本
  • page 默认 8KB，是缓存和读写的基本单位；大 relation 会按 1GB 分段成多个文件

2. Heap Page 与 Tuple

Heap page 大致由页头、行指针数组、空闲空间和 tuple 数据组成：

  • 行指针（item identifier）指向 tuple，同一页内移动 tuple 时索引中的定位信息不必改变
  • tuple header 保存可见性相关信息，如创建/删除它的事务 ID（xmin/xmax）
  • 索引项通常保存键值和 TID（block number + item offset），通过 TID 回表读取 heap tuple
  • 删除不会立刻移除 tuple，而是把它标记为对后续事务不可见，等待 VACUUM 回收

3. TOAST：大字段放在哪

  • 单个 tuple 太大时，PostgreSQL 会压缩或把大字段拆分到关联的 TOAST 表中
  • text、bytea、jsonb 等变长类型可以被 TOAST；读取未使用的大字段时通常不必加载其外部值
  • 大字段频繁读取或更新仍会增加 IO、WAL 和表膨胀，不应把数据库当作无边界对象存储

查询怎么快速定位

索引 + 统计信息 + 成本优化器

1. 常用索引类型

类型 适合场景
B-tree 默认索引；等值、范围、排序、前缀匹配
Hash 纯等值查询
GIN 一个值包含多个元素，如数组、全文检索、jsonb
GiST 几何、范围、近邻搜索等可扩展查询
SP-GiST 可分区搜索空间，如前缀树、四叉树
BRIN 数据与物理顺序强相关的超大表，如按时间追加的数据
  • PostgreSQL 没有聚簇主键；CLUSTER 可按某个索引一次性重排表，但不会持续维护物理顺序
  • 多列 B-tree 索引通常在约束最左侧列时最有效；把等值过滤列放前面，再考虑范围与排序列
  • 部分索引只索引满足谓词的行，例如未删除订单，可减小索引并降低写放大
  • 表达式索引可索引 lower(email) 等表达式，但查询表达式必须与索引定义匹配
  • INCLUDE 可添加非键列形成覆盖索引；Visibility Map 条件满足时可走 Index-Only Scan，避免访问 heap

2. 执行计划

排查慢 SQL 首先使用：

  • EXPLAIN 只显示估算；加 ANALYZE 会真实执行语句，分析写操作时应放在事务中并回滚
  • 重点比较 estimated rows 与 actual rows：偏差很大通常意味着统计信息不足、列相关性未建模或数据分布倾斜
  • BUFFERS 显示命中缓存和实际读写的 page 数，比只看耗时更容易定位 IO 问题
  • 常见扫描：Sequential Scan、Index Scan、Index-Only Scan、Bitmap Index/Heap Scan
  • 常见连接：Nested Loop、Hash Join、Merge Join；没有一种连接算法永远最好

ANALYZE 收集表和列的统计信息供优化器估算。数据分布明显变化后，应确认 autovacuum/analyze 是否及时运行；相关列可使用扩展统计信息改善估算。

并发读写怎么保证正确

MVCC 快照处理读写并发，锁处理写写冲突

1. MVCC 与快照

  • 修改行时创建新 tuple version，并用 xmin、xmax 等信息描述版本的生命周期
  • 查询依据当前 snapshot 和事务提交状态判断 tuple 是否可见，而不是简单读取物理上的最新版本
  • 普通 SELECT 不阻塞普通写操作，普通写操作也不阻塞普通 SELECT
  • 旧版本仍占用空间；长事务会让旧版本持续可能可见，从而阻止 VACUUM 回收

2. 隔离级别

隔离级别 快照行为 说明
Read Uncommitted 按 Read Committed 处理 PostgreSQL 不提供脏读
Read Committed 每条语句获取新快照 默认级别，同事务内两次查询可能看到不同结果
Repeatable Read 第一条非事务控制语句时获取事务级快照 PostgreSQL 还会阻止标准所描述的幻读
Serializable Serializable Snapshot Isolation 检测危险依赖，可能以 serialization failure 中止事务
  • Serializable 不等于给所有读加锁；应用必须能够重试 serialization_failure
  • MVCC 保证可见性，不自动保护跨行、跨表的业务不变量；优先用唯一约束、外键、排除约束等声明式约束

3. 行锁与表锁

显式行锁常见形式：

  • SELECT ... FOR UPDATE：阻止其他事务修改或删除这些行
  • FOR NO KEY UPDATE：比 FOR UPDATE 稍弱，普通 UPDATE 常取得此锁
  • FOR SHARE / FOR KEY SHARE：用于共享读取和外键键值保护
  • NOWAIT：不能立即获得锁就报错；SKIP LOCKED：跳过已锁行，常用于多 worker 任务队列

PostgreSQL 也有多种表级锁模式。普通 DML 会自动取得相应表锁，DDL 常需要更强的锁；例如 ALTER TABLE 的许多形式会请求 ACCESS EXCLUSIVE，上线前必须评估等待和阻塞。

死锁无法完全避免，PostgreSQL 检测到死锁后会中止其中一个事务。降低概率：

  • 多个事务以固定顺序访问对象
  • 缩短事务，不在事务中等待网络或用户输入
  • 确保条件有合适索引，避免锁定超出预期的行
  • 为在线请求配置合理的 lock_timeout 和 statement_timeout

故障后数据怎么恢复

WAL + Checkpoint + Backup

1. WAL

  • PostgreSQL 遵循 Write-Ahead Logging：数据页刷盘前，对应 WAL 必须先持久化
  • 事务提交通常要求其 WAL 刷盘；数据页可以稍后写回，因此提交不需要随机写所有受影响页面
  • 崩溃恢复从最近检查点开始重放 WAL，使数据文件恢复到一致状态
  • WAL 同时用于物理流复制、归档和时间点恢复（PITR）

full_page_writes 在检查点后首次修改页面时记录完整页映像，避免操作系统崩溃造成 torn page。不要为减少 WAL 量随意关闭它。

2. Checkpoint

  • checkpoint 确保其目标位置之前的脏页最终写入数据文件，并记录恢复起点
  • checkpoint 太频繁会增加 IO 和完整页映像数量；太稀疏会延长恢复时间并需要更多 WAL 空间
  • max_wal_size、checkpoint_timeout、checkpoint_completion_target 共同影响检查点节奏
  • 应让写盘尽量平滑，而不是只追求更少 checkpoint

3. 备份与 PITR

方式 特点 适用场景
pg_dump / pg_restore 逻辑备份，可选择对象、跨版本迁移 中小数据库、对象级恢复、升级迁移
pg_basebackup 物理基础备份 整个实例恢复、搭建物理副本
基础备份 + WAL 归档 可恢复到指定时间点 生产级灾备与误操作恢复
  • 复制不是备份：误删和逻辑错误会同步到副本
  • 只有实际做过恢复演练，才能确认备份完整、WAL 连续、密钥和恢复步骤可用

旧版本怎么回收

VACUUM 是 PostgreSQL 正常运行的一部分，不是偶尔执行的清理任务

1. VACUUM 与表膨胀

  • 普通 VACUUM 标记和回收 dead tuple 占用的空间，供同一张表后续复用；通常不会把文件缩小归还操作系统
  • VACUUM FULL 重写整张表并归还空间，但需要强锁且需要额外磁盘空间，不应作为日常维护手段
  • ANALYZE 更新优化器统计信息；autovacuum worker 通常同时承担自动 VACUUM 和 ANALYZE
  • 更新和删除多、长事务多、autovacuum 跟不上时，会出现 heap/index 膨胀、查询变慢和 WAL 增长

2. HOT Update

满足以下条件时，更新可能成为 Heap-Only Tuple（HOT）更新：

  • 没有修改任何索引引用的列
  • 当前 page 有空间容纳新版本

HOT 可以避免为新版本更新所有索引，并允许在 heap page 内沿版本链查找。为高频更新表设置略低的 fillfactor，可预留页内空间提高 HOT 概率，但会牺牲存储密度。

3. Transaction ID Wraparound

  • PostgreSQL 使用有限宽度的事务 ID，并通过模运算判断版本先后
  • 足够老且所有事务都可见的 tuple 必须被 freeze，否则事务 ID 回卷会威胁可见性判断
  • autovacuum 会执行防回卷 vacuum；若长期禁用或无法运行，PostgreSQL 最终会拒绝新事务以保护数据

重点监控：数据库事务年龄、最老事务、最老复制槽、长时间 idle in transaction 的会话，以及 autovacuum 是否被阻塞。

表怎么设计不埋坑

先用数据类型和约束表达业务事实，再按查询模式设计索引

1. 数据类型

场景 推荐 注意事项
整数 integer / bigint 按范围选择；自增可用 identity column
金额 numeric(p, s) 或最小货币单位整数 不使用浮点数表达精确金额
时间点 timestamptz 内部按 UTC 存储，显示时按会话时区转换
本地日期/时间 date / timestamp 仅在语义确实不含时区时使用 timestamp
文本 text PostgreSQL 中 text 与无合理上限的 varchar 通常无性能差异
UUID uuid 比字符串表示紧凑且语义明确
半结构化数据 jsonb 可索引，但不替代稳定核心字段和关系约束
有界集合 array / range / multirange 适合明确场景，不应隐藏需要关联查询的实体关系
  • 优先使用 GENERATED ... AS IDENTITY，而不是旧式 serial 伪类型；identity 的序列依然需要理解权限和显式插入语义
  • NULL 表示未知或不适用，不应仅为省事滥用；可确定的业务条件使用 NOT NULL
  • CHECK、UNIQUE、FOREIGN KEY、EXCLUDE 约束比只在应用层校验更能抵抗并发竞态

2. 主键与索引

  • PostgreSQL 主键不决定表的物理布局，因此随机 UUID 不会像 InnoDB 聚簇主键那样直接引发 heap 页分裂
  • 但随机键仍会降低 B-tree 局部性、增加索引体积和缓存压力；可根据分布式生成、信息泄漏、写入局部性等需求选择 bigint、UUID v7 等方案
  • 每个索引都会增加 INSERT、UPDATE、DELETE 的 CPU、IO 和 WAL 成本；删除重复、未使用的索引
  • PostgreSQL 不会自动为外键引用列创建索引；删除或更新父表键时，缺少该索引可能导致大范围扫描和长时间锁等待

3. JSONB 与关系建模

  • 稳定、需要约束、连接、排序和频繁过滤的字段优先建为普通列
  • 结构变化频繁、不同记录属性差异大的附加信息可放 jsonb
  • 给实际查询路径建 GIN 或表达式索引，不要默认给整个大 JSON 文档创建昂贵的通用索引
  • 将 JSONB 作为逃避 schema 设计的方式，会失去类型、约束、清晰迁移和精确统计信息

4. 分区表

  • 声明式分区支持 range、list、hash 分区，适合超大表的数据生命周期管理和分区裁剪
  • 分区主要解决数据管理、批量删除和特定查询的扫描范围，不是普通性能问题的默认答案
  • 分区键必须出现在分区表的唯一/主键约束中；分区过多会增加规划和维护成本
  • 按时间分区时，删除旧分区通常比批量 DELETE 更快，也更少产生 dead tuple 和 WAL

单点故障与数据增长怎么处理

物理复制保留整个实例，逻辑复制传递选定表的数据变化

1. 物理流复制

  • primary 持续产生 WAL，standby 接收并重放 WAL，得到物理层一致的副本
  • 默认异步复制：primary 提交不等待 standby，故障切换可能丢失尚未复制的事务
  • 同步复制可等待一个或多个 standby 确认 WAL 已写入、刷盘或应用，降低 RPO，但增加提交延迟和可用性耦合
  • hot standby 可承担读查询，但回放 WAL 可能与长查询冲突；长查询也可能间接阻碍 primary 清理旧版本

复制槽可防止 primary 删除消费者尚未读取的 WAL，但消费者长期停滞时会让 WAL 无限积压。必须监控 slot lag，并为失联消费者设置治理策略。

2. 逻辑复制

  • publication 定义发布哪些表的变化，subscription 负责订阅和应用
  • 适合选择性复制、跨版本迁移、数据汇聚和部分零停机迁移
  • 它不是完整实例复制：DDL、序列状态和部分对象需要单独管理；冲突处理也需提前设计
  • 被发布表通常需要合适的 replica identity，才能标识 UPDATE/DELETE 的目标行

3. 扩展路径

  1. 先优化 SQL、索引、统计信息和连接数
  2. 调整实例资源与关键参数，使热点工作集尽量被 OS page cache 与 shared buffers 服务
  3. 用只读副本分担读流量，同时明确读后写一致性要求
  4. 用分区管理超大表
  5. 只有单机写入或容量边界确实成为瓶颈时，再引入应用分片或分布式 PostgreSQL 方案

PostgreSQL 本身不提供透明的内置自动分片；分片会带来跨分片事务、全局约束、再平衡和运维复杂度。

监控与排查

1. 关键观测入口

  • pg_stat_activity：连接、事务状态、当前 SQL、等待事件
  • pg_stat_statements：按归一化 SQL 聚合调用次数、总耗时、平均耗时和 IO 信息
  • pg_stat_user_tables：顺序/索引扫描、live/dead tuple、vacuum/analyze 时间
  • pg_stat_user_indexes：索引扫描次数和读取情况
  • pg_stat_replication / pg_stat_wal_receiver：复制状态和延迟
  • pg_locks：锁持有和等待；需与 pg_stat_activity 联查阻塞链

2. 慢查询排查顺序

  1. 用 pg_stat_statements 找总耗时高或单次耗时高的 SQL
  2. 用 EXPLAIN (ANALYZE, BUFFERS) 对比估算与实际行数、扫描页数和最慢节点
  3. 判断是缺索引、统计偏差、返回数据过多、连接算法不合适、锁等待，还是磁盘/CPU 饱和
  4. 查看 dead tuple、autovacuum、长事务和表/索引膨胀
  5. 修改后比较相同数据规模和负载下的计划与指标，而不是只看一次执行时间

重点告警：

  • 连接数接近 max_connections，或大量连接长期 idle
  • 长事务与 idle in transaction
  • 阻塞链、死锁和 statement timeout
  • dead tuple 持续增长、autovacuum 长期不运行
  • transaction ID age 接近冻结阈值
  • WAL 生成量、归档失败、复制槽 WAL 积压和复制延迟
  • checkpoint 过于频繁、临时文件激增、缓存命中率异常变化
