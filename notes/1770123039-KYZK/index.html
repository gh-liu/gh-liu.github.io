提供"鸭子类型"的能力

  • S: 让不同类型具备同一种能力
  • C: Go 没有传统继承，也不想让类型显式声明 implements
  • Q: Go 如何在没有继承的情况下表达抽象和多态？
  • A: 用 interface 表达类型集合，具体类型只要结构匹配(方法匹配)，就隐式实现接口

Interface 描述一个值在当前上下文中应该具备什么行为

类型集合

如Specification所说：接口类型定义了一个类型集合。接口值可以储存类型集合内任一类型的值。

注意：可以存储类型集合中任一类型的值，这是一种和类型。

接口由一组的方法或类型并集组成，其中类型并集，包含一个或多个类型或底层类型

结构匹配

Go有个方法集 method sets的概念，每个类型都有一个与之关联的方法集合(可能为空)。

接口类型I的方法集合，如果和具体类型T的方法集合一致，那么就是结构匹配的，可以说这个具体类型T实现了这个接口类型I。

这其实也是 Go 接口的优点之一，隐式接口：实现了接口的所有方法的类型就隐式地实现了接口。

在无泛型的时代，使用方法集进行结构的匹配，已经足够了，但是有了泛型...

有泛型后，接口承担起对类型参数的约束，这个类型集约束了类型参数的范围。 类型集中可以是具体类型，也可以是底层类型~Type。详细见泛型。

总结：

  1. 用作普通类型：在运行时装一个具体值，并动态分发方法；这种接口满足方法集合匹配；
  2. 用作泛型参数约束：在编译期约束参数的类型

这里可以大概感觉到：Go 的接口是静态的（编译时检查）和动态的（运行时动态分发）的结合体。也就引出了，结构匹配发生在什么时候？

什么时候进行结构匹配

先思考下，什么时候会需要匹配？把一个具体类型的值，赋值给一个接口值。发生在：

  1. 直接赋值: var i InterfaceType = T{}
  2. 函数接受接口类型值，调用的时候使用具体类型的值作为参数: func(i InterfaceType){}
  3. 函数返回接口值，函数体内部返回具体类型的值: func()InterfaceType{ return T{} }
  4. 断言某个接口值是否实现了接口类型: val, ok := iVal.(InterfaceType)

其中：1/2/3 是编译期能确定的接口实现检查；4 是运行时类型断言。

运行时（动态）

接口值在运行时保存具体动态类型和值

  • 可做类型断言和 type switch 查询实际类型
  • 通过接口方法调用发生动态分发

（编译期）静态

  • 检查赋值、传参、返回值的类型是否满足接口
  • 作为泛型约束，限制类型参数的类型集合

接口值可以保存具体类型的信息（编译期/链接期会把用到的类型描述符放进二进制），可以被反射提供的API获取并使用

类型元数据（如 abi.Type/rtype、方法表等）由编译器为程序中需要的类型生成，并由链接器保留到最终二进制中。 运行时做的是：当具体值被装箱成接口值时，把指向这份类型元数据的指针和指向数据的指针一起放进接口值里。

  • 对 interface{} / any：接口值第一字就是具体类型的类型指针，第二字是数据指针。
  • 对非空接口：接口值第一字是 itab 指针，itab 关联“接口类型 + 具体类型”，其中也能到达具体类型元数据和方法入口；第二字是数据指针。
  • reflect.TypeOf(x) / reflect.ValueOf(x) 接收 any，调用前普通值会先转换成 interface{}：这一步只是把已有类型元数据指针装进接口值；reflect 再在运行时拆开接口值读取它，不会临时生成类型元数据。

运行时表示

接口值由两个字组成：

┌─────────────────┬─────────────────┐
│   itable 指针   │    数据指针     │
└─────────────────┴─────────────────┘

itable 包含：

  1. 类型元数据

  2. 函数指针列表

itable 对应的是接口类型 + 动态具体类型这一对组合，而不是只对应接口类型。

any

any 是 interface{} 的别名，无方法，运行时表示里没有 itab，第一字直接保存具体类型指针：

┌──────────────┬──────────────┐
│  类型指针    │  数据指针    │
└──────────────┴──────────────┘

方法动态派发

调用接口的方法，在编译期间不能确定接口的具体类型（Go: 编译器去虚化 devirtualize），那么在运行期间决定调用该方法的哪个实现；

这就意味着，接口方法的调用，可能是：

  1. 具体类型调用，编译期确定；使用 -N 关闭编译器优化

  2. 接口类型调用，运行时动态派发：获取 itab->fun[k] 到寄存器，iface 的 data 作为接收者参数放到寄存器，通过 CALL 间接调用

设计缺陷？

  • 未初始化的接口值是 nil
  • 但如果接口值里装的是具体类型 + 该类型的 nil 值，那么这个接口值本身不为 nil，必须两个都为nil才行

最佳实践

  1. Accept interfaces, return concrete types.
  2. Small interfaces compose better.
