agent

1. 别什么都用 agent

  1. 搜索空间大吗？
  2. 单次任务价值高吗？
  3. 模型核心能力够吗？
  4. 错误能被快速发现吗？
  5. agent 有足够观测信息吗？

4 个问题筛场景：

  • 复杂度：能画决策树 → workflow；模糊问题空间 → agent
  • 价值：能 cover token 成本才用 agent
  • 关键能力：主路径每一步模型都得能做
  • 错误成本：贵且难发现 → 加 read-only / human-in-the-loop

典型甜蜜点：coding（模糊 + 高价值 + 模型强 + test/CI 易验证）

2. 尽可能简单

agent = 模型在循环里使用工具，只有 3 个组件：

  • 环境
  • 工具
  • system prompt

先把这 3 个迭代到位；缓存 / 并行 / 进度展示等行为稳定后再加。

3. 用 agent 的视角想

把自己塞进 10-20K 的 context window：只有 system prompt + 工具描述 + 最近几个 observation。

做一遍这个练习就知道 agent 缺什么（环境信息 / 推荐动作 / 进度状态）。

偷懒招：把 system prompt、工具描述、trajectory 丢给 Claude，让它说哪里有歧义、缺什么参数、为什么这么决定。

3 个开放问题

  1. budget-aware：怎么给 agent 定 token / 时间 / 钱的预算
  2. self-evolving tools：agent 自己设计 + 改进工具
  3. multi-agent：从同步回合制扩展到异步通信
