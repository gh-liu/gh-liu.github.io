连续栈： 给每一个goroutine的栈分配一段连续的空间，当空间填满时，使用reallocation/copy增长空间。 为什么： 1.当前分裂栈机制有一个”hot split”问题-如果栈差不多满的时候，一个调用将会强制生成一个新栈块。当调用返回的时候，这个新栈块就释放掉了。如果这个调用重复发生在一个紧密的循环中，这个调用产生的分配/释放开销会造成十分明显的开销。 2.有拆分stack时候，stack分配/取消分配工作永远不会完成-每当stack大小在任一方向上超过阈值时，都需要进行额外的工作。 使用连续栈，我们解决了两种方式，stack增加到所需的大小，当不在被工作需要的时候就收缩。 如何： 你怎么可以移动一个stack？结果是编译器的逃逸分析提供了一个非常重要的不变值：stack上指向数据的指针，其只在调用树中传递。任何其他的逃逸（写到全局变量，返回至父，写至堆等。）阻止这个数据分配到栈上。这个不变量意味着指向stack数据的唯一指针在stack本身中（下面有意外），其使的我们可以轻易找到并修改当我们需要复制这个栈的时候。 溢出检查： 栈溢出检查与分裂栈的溢出检查相似，主要的不同是我们不再需要知道参数大小，因为不在需要复制参数到一个新栈段内。这在一定程度上简化了拓展stack的goroutine族的激增，并且消除了大部分对nosplit指令的需求。 我们甚至也不需要知道帧的大小。只要将当前栈大小乘2即可，如果依旧不能满足，将会促发栈溢出检查并再次将栈乘2。重复直至满足栈大小需要。 复制： 当栈溢出检查促发时： 分配一个比旧栈大小稍大的新栈。我们应该以指数增长大小的方式分配（限制已经完成的复制工作），并可能会四舍五入为一个不错的大小的倍数（因此分配器可以变的更加有效）。！！我目前正在计划使用两种大小的幂，并且将每次重新分配加倍。 复制旧的栈到新的栈，栈被视作为一个字节指针的数组，且每一个指针从旧栈复制到新栈，每个字节指针与旧栈的边界进行比较，如果在范围内，将会调整指针相同偏移量从TOS到新栈。 只有真的指针才可以被调整 - 我们需要避免整型看起来像指针。所以我们需要一个数据结构，其可以告诉我们，在栈上每一个指针大小的word，是否是个指针。幸运的是，Carl正在生成一个用于精确GC的数据结构。 所有从外进入到栈的指针需要被调整。尽管有逃逸分析，但是还是有一些，包括： 延迟调用（deferred）的函数对象； 延迟调用（deferred）的参数； 延迟调用（deferred）的函数对象的闭包参数； 将要传递到溢出帧的函数对象（在寄存器中传递，而不是在stack中传递，至少在x86中是这样） `传递到溢出帧的函数对象的闭包参数 相同的规则试用于复制的时候，我们需要知道这个数据确实是一个指针，并且如果是的话，知道其是否指向旧栈 Reflect.call: Reflect.call调用一个函数，其被给与一个指向参数区域与参数大小的指针。旧的栈分裂机制，强制生成一个栈块，所以这个可变大小的参数区域可以分配到栈的顶部。新机制需要分配这些参数在栈的任何地方，为了模拟一个可变大小帧，我们定义了一系列2次幂大小的固定大小的帧的函数，其可以用来做跳板。

    reflect.call(function, args, argsize)

            if argsize <= 16: jump reflect.call16(function, args, argsize)

            if argsize <= 32: jump reflect.call32(function, args, argsize)

            if argsize <= 64: jump reflect.call64(function, args, argsize)

    …


    reflect.call16(function, args, argsize)

            sp -= 16

            copy [args, args+argsize] to [sp, sp+argsize]

            call function

            copy [sp, sp+argsize] to [args, args+argsize]  (for return values)

            sp += 16

            return

我建议reflect.call的参数大小最大为64k 复制器需要知道哪些是这些帧的指针，哪些不是。这个信息可以通过查看被调用函数的签名来获得。 (and for vararg calls also the argument size?对vararg调用也是这个参数大小么？) 栈追踪： 栈追踪在一定程度上简化了，因为其不在需要处理栈上的分裂了。 GC： 和栈追踪差不多，GC对一个g的栈扫描也得到了简化。 Defer/panic/recover： 这些机制的实现有轻微的变化，并且大多数实例得到了简化。旧机制使用栈分裂点去标记哪里发生了panic。 测试： 对于测试来说，所有的栈将会分配到堆外的某处。当一个栈复制完成，旧栈将会munmap(),所以对旧栈的错误范围会被trap。 CGo： 不确定，我目前还不知道问题是什么。 Shrinking： 当一个goroutine在其生命周期的开始使用了许多栈空间，但是在剩下的生命周期使用了很少的栈空间，就会有很多空闲的栈空间。我们需要使用某种方式回收这些空闲的空间，想法： 在GC的时候，如果一个goroutine使用未超过其总共栈空间的X%，复制其到一个小一点的栈。当其需要更多空间的时候，就会拓展栈空间。 在GC的时候，如果一个goroutine使用未超过其总共栈空间的X%，降低其stack guard的值，如果在下次GC的时候没有触发到guard，就复制其到一个小一点的栈。 我当前的计划是在GC的时候，!!如果goroutine最多使用了其stack的¼，则释放栈底部的½。如果stack足够大，我将直接释放它，否则复制。 Starting stacks: 所有的g默认以最小的栈空间开始。当一个G完成时，我们释放它的stack，因为它可能很大，而不是恰好在该G上启动的下一个go例程所需要的(也许我们使用栈收缩代码处理这里)。 我们可以保留每个go例程使用多少stack的统计信息（由函数地址键控），并在启动go例程之前预先分配这些stack。每一次一个g执行完成时，统计信息可以被更新。这不是完美的，例如一个goroutine栈的使用依赖于数据，！！但这可能是一个适当的起点。 如果我们想要在一个g中记录栈的最大使用情况，这个机制同样适用与分裂栈。然而，其必须更加保守，在拆分裂栈中，分配太大的初始栈是无法撤消的。 Disadvantages： 更多的虚拟内存压力/碎片。尤其是很大的栈空间的时候，找到一个连续内存空间放入栈变成非常困难。可能的话，一些混合模式可以做到更好，举例来说，复制栈空间到1Mb，在后面可以分裂它。这可能有帮助，但是我不认为这值得额外的实现复杂性去获得这两种机制。 进入栈的指针的限制。所有进入栈空间的指针需要被严格的解释。目前，我们的逃逸分析，确保栈中没有杂散指针。将来也许会有一些场景，我们希望允许一些指针进入栈，但这个实现可能会阻止。我还不知道这些场景会是什么，但是这个限制值得注意。 可能需要对运行时的C/asm代码进行审核，以确保栈指针没有任何问题。
