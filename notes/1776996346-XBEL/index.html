S: 程序和系统都会随时间演化，路径很多，难以逐条推理。 C: 代码可能符合经验法则、看起来正确，但边界条件仍然错。 Q: 如何用一种稳定的方式推理动态变化中的程序和系统？ A: 找出 invariant：演化过程中始终成立的性质。它压缩了时间维度，让我们不用枚举所有路径，而是围绕共享约束写代码、设计系统。

小尺度：用 invariant 写对细节

对 insertion point 来说，目标是返回最小的索引 i，使得把 x 插入 xs[i] 后数组仍然有序。

可以把循环 invariant 明确写成：

也就是：

  • lo 左侧的元素一定小于 x。
  • hi 右侧的元素一定大于等于 x。
  • 答案始终落在 [lo, hi] 里。

于是分支应该贴合 invariant：

xs[mid] < x 时，mid 也属于“小于 x 的左侧区域”，所以下界可以推进到 mid + 1。否则，mid 可能就是第一个大于等于 x 的位置，所以上界只能收缩到 mid。

原始写法的问题是：当 x == xs[mid] 时，把 lo 设为 mid 会破坏“lo 左侧都小于 x”这个 invariant。

大尺度：用 invariant 约束系统演化

系统级 invariant 是一些紧凑、会影响整个代码库的设计约束。

例子：

  • Cargo：用户输入的文件路径保持原样，不随意 canonicalize。因为文件系统语义复杂，看似等价的路径未必真的等价。
  • rust-analyzer：IDE API 暴露的是一个完整、冻结的世界快照；内部可以按需计算和缓存，但调用方不用感知这些细节。
  • TigerBeetle：启动后不再分配内存；交易逻辑不直接读磁盘；读磁盘 API 对上层表现为不会失败。

这些 invariant 一旦成立，很多局部代码就不用反复处理同一类复杂性。代价是它们会“传染”到系统各处，要求所有代码都遵守。

记忆点

  • Invariant 是“变化过程中不变的东西”。
  • 小尺度 invariant 帮助写对循环、递归、状态机等细节。
  • 大尺度 invariant 帮助大型系统保持简单、统一的心智模型。
  • 好的 invariant 应该足够精确，能指导代码分支和边界条件。
  • 如果某段代码需要靠反复试错 +1、-1 或特殊分支修补，通常说明 invariant 还没想清楚。
