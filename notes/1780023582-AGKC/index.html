sing-box 是一个通用代理平台（universal proxy platform），不是单一协议客户端，而是一个配置驱动的网络代理、DNS、路由、转发引擎。

核心抽象

最核心的抽象是：Inbound / Route / Outbound。

  • Inbound：流量入口，负责接收或接管流量。
    • 例如 socks、http、mixed、tun、redirect、tproxy。
    • 也可以作为代理服务端接收 shadowsocks、vmess、vless、trojan、hysteria2 等协议。
  • Outbound：流量出口，负责把流量送出去。
    • 例如 direct、block、socks、http、shadowsocks、vless、trojan、wireguard、tor、ssh。
    • 也支持组合出口，例如 selector、urltest。
  • Route：分流决策层，根据规则选择 outbound。
    • 可按域名、IP、端口、协议、入站 tag、进程、GeoIP、Geosite、rule-set 等匹配。
  • Rule / Rule Set：规则抽象。
    • Rule 是单条匹配逻辑。
    • Rule Set 是可复用、可远程更新的规则集合。
  • DNS：域名解析系统，既服务代理内部解析，也可以处理客户端 DNS 请求。
    • 支持 DNS rules、多 DNS server、缓存、FakeIP、IPv4/IPv6 策略、EDNS Client Subnet 等。

数据流

典型数据流：

过程：

  1. 应用发起连接，例如访问 example.com:443。
  2. 流量进入某个 inbound：
    • 应用显式配置 SOCKS/HTTP 代理；
    • 或系统流量被 TUN 接管；
    • 或 Linux 透明代理由 iptables/nftables 转入 redirect / tproxy。
  3. sing-box 提取连接元信息：目标域名/IP、端口、网络类型、来源 inbound、进程信息等。
  4. 如目标是域名，可能进入 DNS 子系统，按 DNS rules 选择解析服务器、使用缓存或 FakeIP。
  5. route 按规则匹配，得到 outbound tag。
  6. outbound 执行转发：
    • direct：直连目标；
    • block：阻断；
    • 代理协议 outbound：封装后发给代理服务器；
    • selector：转给当前选择的子 outbound；
    • urltest：按延迟测试选择出口。
  7. 返回流量沿原路径回到应用。

TUN 与透明代理

TUN 和透明代理都能让应用无需显式配置代理，但接入层不同。

维度 TUN inbound 透明代理：redirect / tproxy
接入方式 创建虚拟网卡，系统路由把流量送入 TUN 用 iptables/nftables 拦截流量
OS 视角 类似 VPN 类似内核防火墙规则劫持
配置复杂度 相对简单 更复杂，需要防火墙、mark、策略路由
UDP 支持 更自然 redirect 不适合复杂 UDP，tproxy 更完整
适合场景 桌面端、移动端、全局代理 Linux 网关、路由器、旁路由

一句话：TUN 是从虚拟网卡/路由层接管流量；透明代理是从防火墙/策略路由层劫持流量。

Rule Set

sing-box 支持 route.rule_set，规则里可以通过 rule_set 引用本地或远程规则集。

Rule set 类型：

  • inline：直接写在配置里。
  • local：引用本地规则文件。
  • remote：从 URL 下载规则文件。

Rule set 格式：

  • source：sing-box 的 JSON source 格式。
  • binary：编译后的 .srs 格式。

注意：sing-box 不直接支持 Clash rule-provider YAML 格式，需要转换成 sing-box source JSON 或 .srs。

常见 .srs 规则来源：

  • SagerNet/sing-geosite
  • SagerNet/sing-geoip
  • MetaCubeX/meta-rules-dat
