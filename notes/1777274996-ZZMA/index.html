geektime 周志明的架构课

核心观点

这门课的重点不是某个框架怎么用，而是建立一张完整的现代架构地图：系统为什么会从单体走向微服务，再走向云原生；每一步解决了什么问题，又引入了什么复杂度。

  • 架构演进的本质是复杂度转移：从单体到 SOA、微服务、云原生，不是复杂度消失，而是复杂度从代码内部转移到服务间通信、运维治理和基础设施层。
  • 微服务不是目标，而是手段：它解决的是组织规模、业务边界、交付效率的问题，同时也带来事务、一致性、治理和观测上的新成本。
  • 分布式系统真正难的不是拆服务，而是拆完以后怎么保证可靠性：远程调用、事务、缓存、服务发现、容错、限流、安全、可观测性，才是长期成本所在。
  • 云原生的核心价值是把复杂度下沉到基础设施：容器、Kubernetes、服务网格、不可变基础设施，本质上都在把分布式问题标准化、自动化、平台化。
  • 架构师的工作不是追热点，而是在约束下做取舍：在成本、性能、可靠性、扩展性和团队协作之间找到合适平衡。

主线

1. 演进中的架构

课程先从 Unix、单体、SOA、微服务，一路讲到后微服务和无服务架构。

这一部分的重点不是背定义，而是理解：

  • 为什么旧架构在当时是合理的
  • 新架构是为了解什么问题而出现的
  • 新方案引入了哪些新的代价和边界

可把它看成一条认知主线：

单体 -> 服务化 -> 微服务 -> 云原生 -> Serverless

2. 架构师的视角

课程把"架构设计"从抽象概念拆成一组可讨论的问题：

  • 远程调用怎么做
  • RPC 和 REST 怎么选
  • 本地事务和分布式事务如何处理
  • 缓存如何协同
  • 认证、授权、凭证、加密、传输安全如何落地

这一部分的价值在于：架构设计不再是拍脑袋选框架，而是围绕一系列约束条件做工程判断。

3. 分布式的基石

这是整门课最通用的一部分，覆盖现代分布式系统的共性问题：

  • 服务发现、路由、负载均衡
  • 故障处理、容错、限流
  • 分布式共识
  • 零信任安全
  • 日志、追踪、度量等可观测性能力

这部分传达的核心认识是：

分布式系统的难点，从来不在"把模块拆开"，而在"如何让多个不可靠节点对外表现得足够可靠"。

4. 从微服务到云原生

这里的关键概念是不可变基础设施。

课程的判断很明确：云原生不只是容器化，也不只是 K8s，而是把原本需要应用自己承担的一部分复杂度，转交给基础设施和平台层处理。

重点包括：

  • 容器：封装运行环境
  • Kubernetes：调度、编排、资源管理
  • 网络与存储：为分布式运行提供统一底座
  • 服务网格：把通信治理从业务代码里剥离出来

一句话概括：

微服务主要解决服务拆分问题，云原生主要解决拆分之后的交付、治理和运行问题。

5. 探索与实践

课程最后用 Fenix's Bookstore 做多种架构实现和部署实践，包括：

  • 基于 Spring Boot 的单体架构
  • 基于 Spring Cloud 的微服务架构
  • 基于 Kubernetes 的微服务架构
  • 基于 Istio 的服务网格架构
  • 基于云计算的无服务架构

这部分适合把前面的抽象概念映射回真实工程，观察同一个业务系统在不同架构下的收益与代价。

这门课最值得吸收的判断

  1. 单体不是落后方案：在团队规模、业务复杂度、交付频率还没到一定程度前，单体往往是性价比最高的选择。
  2. 微服务的收益和成本必须一起看：拆分提升了自治和扩展性，也放大了网络调用、数据一致性和运维治理问题。
  3. 可靠性是分布式架构的第一主题：容错、隔离、限流、观测、安全，比"怎么拆"更决定系统长期可维护性。
  4. 云原生的价值在于标准化和自动化：它让很多高复杂度能力不再由业务团队重复造轮子。
  5. 架构演进没有银弹，只有取舍：任何方案都在交换复杂度、成本和灵活性。

适合怎么读

  • 第一遍通读，建立概念地图，不追求全懂。
  • 第二遍按问题回查，比如事务、缓存、限流、可观测性。
  • 第三遍结合实战章节看同一系统在不同架构下的实现差异。

一句话总结

这门课最大的价值，是帮你建立一种判断力：什么时候该用单体，什么时候该拆成微服务，什么时候该借助云原生基础设施，以及每一步你到底在交换什么。
