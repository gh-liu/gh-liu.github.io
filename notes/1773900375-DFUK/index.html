数据怎么组织和查询

关系表 + SQL

1. SQL 执行流水线

  • 连接器 → 分析器 → 优化器 → 执行器 → 存储引擎（InnoDB）
    • 连接器：你是谁？有什么权限？
    • 分析器：SQL 合不合法？
    • 优化器：怎么执行最快？得到执行计划
    • 执行器：按计划调引擎接口
    • 存储引擎是插件式的，默认且最重要的是 InnoDB

2. 存储：数据放在哪？

  • 按 表空间 → 段 → 区 → 页 → 行 的层级组织。
    • 页（默认 16KB）是磁盘与内存交互的基本单位
  • MySQL 不是一行一行读磁盘的，而是一页一页读的；性能核心指标是读了多少页，而非多少行
  • InnoDB 在内存中维护 Buffer Pool 缓存热点页，避免每次都访问磁盘（缓存数据页，减少磁盘 IO）

2.1 磁盘结构

  • 系统表空间、独立表空间 *.ibd、undo 表空间、redo log 文件、临时表空间
  • 双写缓冲（doublewrite）：先把页写到双写区域再写到真正位置，避免写一半宕机导致页损坏

2.2 页内部结构

  • 文件头/尾（页类型、校验、前后页指针）→ 页头（槽数、记录数、空闲空间）→ 用户记录（数据行，链表串联）→ 页目录（分组建槽，二分查找）
  • 页满了要分裂 → 随机主键频繁触发页分裂；页内链表 → 插入删除不需要移动数据

2.3 行格式

  • 常见格式：Compact、Redundant、Dynamic（默认）、Compressed
  • 每行除用户列外有三个隐藏列：DB_ROW_ID（无主键时自动生成）、DB_TRX_ID（事务 ID，MVCC 用）、DB_ROLL_PTR（指向 undo log，回滚 + MVCC 用）
  • 行溢出：TEXT/BLOB/大 VARCHAR 超过页大小时，行内保留部分数据 + 指向溢出页的指针

查询怎么快速定位

索引

1. 索引：怎么快速找到数据？

  • B+ 树索引：扇出大、树矮（通常 2~4 层）、叶子串链表支持范围查询。哈希只能等值查询不支持范围；二叉树类树太高 IO 多
  • 聚簇索引：叶子存整行数据，通常就是主键索引，一张表只有一个
  • 二级索引：叶子存索引列 + 主键值，查整行要回表到聚簇索引再查一次

索引使用原则

  • 最左前缀：联合索引 (a, b, c) 只能按 a → ab → abc 的顺序利用；遇到范围查询后后续列通常难继续充分利用
  • 覆盖索引优于回表；联合索引通常优于多个单列索引的 index merge
  • 建索引优先考虑：过滤列、排序列、分组列、区分度高的列、更短的列类型；优先扩展已有联合索引而非盲目新增

2. 优化器：有多条路时走哪条？

  • 优化器做两个决定：走哪个索引 和 多表先查谁；基于成本模型（IO 成本 = 读多少页 + CPU 成本 = 过滤/比较多少行）
  • "用了索引"不等于快——回表太多可能还不如全表扫描；统计信息不准时执行计划可能变差
  • 排查慢 SQL 第一步：EXPLAIN 看执行计划

单表访问方式（从好到坏）：

类型 含义
const 主键/唯一索引等值命中一条记录
ref 普通索引等值匹配
range 索引范围扫描
index 扫描整个索引树
ALL 全表扫描

多表连接：

  • 选一个驱动表，用驱动表结果去匹配被驱动表；
    • 优化关键：驱动表尽量小、被驱动表有索引、连接顺序由优化器按成本选择

并发读写怎么保证正确

事务

1. 事务：多人同时读写怎么办？

  • ACID：原子（全做或全不做）、一致（数据始终合法）、隔离（事务互不干扰）、持久（提交不丢失）
  • 隔离级别从松到严：Read Uncommitted → Read Committed → Repeatable Read → Serializable。InnoDB 默认 Repeatable Read
  • MVCC：每行数据保留多个版本，普通 SELECT 读"某个时间点可见的版本"而非当前最新物理值，读不需要加锁，读写不互斥

版本链与 Read View

  • 每行的 DB_TRX_ID 记录最近修改的事务 ID，DB_ROLL_PTR 指向 undo log 旧版本，多次修改形成版本链
  • 事务做一致性读时生成 Read View（当前活跃事务 ID 列表、最小活跃事务 ID、下一个待分配事务 ID），沿版本链找第一个可见版本返回
  • RC vs RR：RC 每次 SELECT 生成新 Read View → 能看到已提交最新值；RR 事务开始时生成一次，后续复用 → 同事务内多次读结果一致

2. 锁：写冲突怎么解决？

  • 锁是分层的：全局锁 > 表级锁 > 行级锁（最常接触）；行锁不是按物理行号加锁，而是锁索引记录
  • 间隙锁锁住两条记录之间的"空隙"，防止别人插入新行（防幻读）；死锁是常见现象，InnoDB 会检测并回滚一个事务

表级锁

  • MDL（元数据锁）：防止"查表"和"改表结构"互相冲突
  • 意向锁：加行锁前先加意向锁，告诉系统"表中某些行将被锁"

行级锁类型

类型 锁什么 作用
记录锁 Record 已有的索引记录 防止修改/删除
间隙锁 Gap 两条记录之间的区间 防止插入（防幻读）
邻键锁 Next-Key 记录锁 + 间隙锁 RR 级别的默认行锁
插入意向锁 间隙中的某个位置 允许不同位置并发插入
自增锁 AUTO_INCREMENT 计数器 保证自增值连续

Next-Key Lock = Record Lock + Gap Lock，是 RR 下部分解决幻读的关键。降低死锁概率：固定访问顺序、走索引缩小锁范围、最易冲突操作放事务最后。

故障后数据怎么恢复

日志

日志 谁的 一句话
redo log InnoDB 崩溃后重做，保持久性
undo log InnoDB 回滚 + 构造 MVCC 版本链
binlog Server 主从复制 + 数据归档
  • redo log：物理日志，记录"某页某偏移做了什么修改"。WAL 思想——先顺序写日志再异步刷数据页，顺序写比随机写快得多
  • undo log：记录修改前的旧值，用于回滚和构造 MVCC 版本链。删除通常先打 delete mark，由 purge 线程异步清理
  • binlog：逻辑日志，记录"这条语句/这个事务做了什么"，用于主从复制、数据归档、时间点恢复

内存 Log Buffer：缓存 redo log

两阶段提交

redo log 和 binlog 分属不同层，只写一个没写另一个主从数据就不一致。两阶段提交保证一致：

1. redo log 写入 prepare
2. binlog 写入
3. redo log 标记 commit

崩溃恢复判断：redo prepare + binlog 完整 → 提交；否则 → 回滚。

表怎么设计不埋坑

Schema 设计

1. 数据类型选择

够用的前提下，越小越好、越简单越好、避免 NULL。

场景 推荐 避免 原因
整数 INT / BIGINT 按范围选 全部用 BIGINT 浪费空间，索引更大
小数/金额 DECIMAL FLOAT / DOUBLE 浮点有精度问题
时间 DATETIME 或 TIMESTAMP 字符串存时间 无法做时间运算和比较
短文本 VARCHAR(N) N 按实际上限 VARCHAR(65535) 内存临时表按声明长度分配
长文本/大对象 TEXT / BLOB 频繁查询的列用 TEXT 可能行溢出，影响 Buffer Pool 效率
  • NULL 的代价：索引、统计、比较需额外处理；COUNT(col) 不计 NULL，COUNT(*) 计全部——容易出 bug。除非业务语义确实需要"未知"，否则 NOT NULL DEFAULT ...

2. 范式与反范式

  • 1NF 列不可再分；2NF 消除部分依赖；3NF 消除传递依赖
  • 先按第三范式设计保证正确性，再针对高频查询做有控制的反范式：冗余列避免高频 JOIN、汇总表避免实时聚合；代价是更新时需同步维护冗余数据

3. 主键设计

稳定、短、递增——稳定（值不变否则所有二级索引都要改）、短（存在每个二级索引叶子里）、递增（顺序插入减少页分裂）。

方案 优点 缺点
AUTO_INCREMENT 递增、短、简单 分布式环境需额外协调
雪花 ID 趋势递增、全局唯一 8 字节、依赖时钟
UUID v4 全局唯一、无需协调 36 字节、完全随机、页分裂严重

4. Online DDL（数据定义预览）

算法 过程 是否锁表
INSTANT 只改元数据 不锁
INPLACE 引擎内部重建 大部分不锁（允许并发 DML）
COPY 创建临时表、拷贝数据 锁表

大表变更建议：优先 INSTANT；不支持则用 gh-ost（基于 binlog 的无触发器方案）。

单点故障与数据增长怎么破

复制

1. 复制

主库写 binlog，从库拉取并重放。

模式 流程 数据安全 性能
异步复制 主库写完 binlog 即返回 主库宕机可能丢数据 最好
半同步复制 至少一个从库确认收到 binlog 才返回 几乎不丢 略慢
组复制 (MGR) Paxos 协议多数派确认 不丢 更慢，但支持多写
  • 复制延迟常见原因：从库单线程回放、大事务、DDL 阻塞。缓解：并行复制（slave_parallel_workers）、拆大事务、避免从库做大查询

2. 读写分离与分库分表

  • 读写分离：主写从读，代价是复制延迟、可能读旧数据
  • 分库分表：水平拆（按行）或垂直拆（按业务域），代价是跨库 JOIN、分布式事务、全局主键

3. 监控与排查

关键指标：QPS/TPS、慢查询数、线程连接数、Buffer Pool 命中率（< 99% 需关注）、复制延迟（Seconds_Behind_Master > 1s 需排查）。慢查询排查流程和 EXPLAIN 字段见 @CS_CLI++mysql.md。
