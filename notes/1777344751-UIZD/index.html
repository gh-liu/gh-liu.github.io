数据如何组织、如何访问、并发时如何保证正确、故障后如何恢复、突破单机边界

对于一个数据库系统，最重要的就是理解其读写路径，以及其核心抽象在这个读写路径上各自起到什么作用； 一条sql的执行，从解析、到优化、到执行，解析/执行没什么好讲的，优化可以讲讲如何快速定位数据，这里就可以想到索引，索引加速数据的定位获取； 以上是一个连接执行一条sql，如果是多个连接，各自执行一条sql，查询改动涉及的数据有重复，这里如何保证互不影响呢？这里就可以想到事务了，事务在并发读写的情况下保证正确型； 还有就是，执行过程中，系统崩了，应该如何处理？如何恢复数据？ 最后就是，如何破解单点的性能瓶颈，支持系统横向拓展。

层次 核心问题 核心机制
单请求执行 一条 SQL 如何运行 解析器、优化器、执行器
数据组织与访问 如何快速找到数据 页、记录、B+ Tree、索引、缓存
并发正确性 多个请求如何互不破坏 事务、锁、MVCC、隔离级别
故障恢复 崩溃后如何恢复正确状态 WAL、Redo、Undo、Checkpoint
分布式扩展 如何突破单机限制 复制、共识、分片、分布式事务

一次数据变化，如何从 SQL 进入内存、被并发控制、被日志记录、最终持久化，并进一步复制到其他节点。

mysql postgresql

数据库问题通常分 4 类：

  1. 查询慢 → 索引 / SQL 优化 / 缓存 / 读副本

  2. 读压力大 → 缓存 / 读写分离 / CDN / 搜索引擎

  3. 写压力大 → 分片 / 异步化 / 批量写 / 消息队列

  4. 数据量太大 → 分区 / 分片 / 归档 / 冷热分离

数据库扩展：Indexing / Replication / Partitioning / Sharding

Indexing： 让单机查询更快。 Replication： 复制多份，扩读和容灾。 Partitioning： 把大表在一个数据库系统内部拆小。 Sharding： 把数据拆到多个数据库实例，扩写和扩容量。
