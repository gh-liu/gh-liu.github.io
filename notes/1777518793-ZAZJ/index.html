一句话补充如下点：

  • 是什么（What）— 一份「优秀 Git commit message」的写作规范，由 7 条具体规则组成，把提交信息当作向协作者（包括未来的自己）传达变更上下文的正式文档来对待。
  • 为什么（Why）— diff 只能告诉你改了什么，只有 commit message 能讲清为什么改；维护者最强大的工具之一就是项目的 git log，结构化、风格一致的提交历史让 blame / revert / rebase / log / shortlog 等命令真正发挥威力，也降低重建代码上下文的成本。
  • 怎么做（How）— 严格遵循 7 条规则；正文聚焦 what & why，不写 how（代码自身解释 how）；不能一句话讲清的就用编辑器写完整正文，避免 -m 草草了事；提交太大写不出 50 字标题时，说明该拆成原子提交。

核心观点：

A diff will tell you what changed, but only the commit message can properly tell you why. 一条精心打磨的 commit message 是开发者向协作者传达变更上下文的最佳方式，也是衡量一个开发者是否是好协作者的标志。

启发点（关键洞察）：

  1. 七条规则（The Seven Rules）：

    1. Subject 与 body 之间用空行分隔（很多 Git 工具靠这个空行来识别标题与正文）。
    2. Subject 限制在 50 字以内（GitHub 超过 50 警告、超过 72 截断；写不下说明提交粒度过大，应拆分为原子提交）。
    3. Subject 首字母大写。
    4. Subject 末尾不加句号（节省宝贵字符）。
    5. Subject 使用祈使语气（imperative mood），与 Git 自身（merge、revert、PR merge）生成的信息风格保持一致。
    6. Body 在 72 字处手动换行（Git 不自动换行，留出缩进余量保证总宽 ≤ 80）。
    7. Body 解释 what & why，而不是 how（how 由代码本身说明，复杂逻辑用源码注释而非 commit message 解释）。

  2. 祈使语气的判断公式：一条合格的 subject 必须能补全这句话—— "If applied, this commit will ___"。

    • ✅ Refactor subsystem X for readability
    • ✅ Remove deprecated methods
    • ❌ Fixed bug with Y / Changing behavior of X / More fixes for broken stuff

  3. 不是每个 commit 都需要 body：简单到一目了然的改动（如 Fix typo in introduction to user guide）一行足矣，可以放心用 -m；但只要值得解释上下文，就该用编辑器写完整正文，不要被 -m 的便利绑架。

  4. 提交历史的「破窗效应」：因为历史混乱所以没人看，因为没人看所以越来越混乱——这是个恶性循环；反过来一份被精心维护的 log 会让 blame / revert / rebase / shortlog 真正活起来，code review 也变成可独立完成的有价值工作。

  5. 聚焦 why 的写法套路：解释「改之前是怎么工作的、有什么问题、现在怎么工作、为什么选这个方案」，而不是逐行复述 diff；issue / PR 编号等元数据放在正文末尾，例如 Resolves: #123 / See also: #456, #789。

  6. 样板模板（可作为编辑器 commit template）：

    Summarize changes in around 50 characters or less

More detailed explanatory text, if necessary. Wrap it to about 72
characters or so. Explain the problem that this commit is solving.
Focus on why you are making this change as opposed to how (the code
explains that). Are there side effects or other unintuitive
consequences of this change? Here's the place to explain them.

- Bullet points are okay, too
- Use a hyphen or asterisk, preceded by a single space

Resolves: #123
See also: #456, #789

  7. 风格一致性比规则本身更重要：规则的目的是「消除猜测」（remove the guesswork），让 log 成为一份真的会被读的文档；起初是负担，习惯后会变成团队的生产力与骄傲来源。
