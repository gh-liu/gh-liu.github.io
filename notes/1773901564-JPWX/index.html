核心概念是 change；一个跨 rebase 不变的唯一标识

不存在”改了但还没 add”这种中间态，working copy 本身就是一个 change

jj describe 对任意 change 修改描述

jj new 把当前 change 定格

jj commit = jj describe + jj new

jj edit kx 编辑指定change，改完后后续自动rebase；推送到远端的 change 是 immutable

jj rebase 移动 change

jj split 拆分当前change

jj commit 执行N次，提交N个空提交，然后一个一个edit后，跳到下一个

jj workspace 工作空间

jj undo 撤回上一个操作

快速入门

jj 和 Git 最大的区别：working copy 本身就是当前变更 @，通常不需要 git add。

最小工作流：

说明：

  • jj st 看当前工作区状态
  • jj log 看变更图；@ 是当前 working-copy change
  • 改完文件后，下一次执行 jj 命令时会自动 snapshot working copy
  • jj commit 默认会提交当前 change，并在上面新建一个空 change 继续工作

常用命令

速记：

  • jj st：看当前状态
  • jj log：看提交图；默认偏向显示 mutable revisions
  • jj log -r ::：看全部历史
  • jj diff：看当前 working copy 相对父变更的 diff
  • jj show <rev>：看某个 change 的描述和内容
  • jj desc：改描述，不改变内容
  • jj new：新建空 change，并切过去继续编辑
  • jj commit：提交当前 change，然后自动创建下一个空 change
  • jj edit <rev>：切到某个已有 change 上继续改
  • jj squash：把当前 change 合并进父 change
  • jj split：把一个 change 拆成两个
  • jj rebase -d <rev>：把当前或指定 change 挪到别的父节点下
  • jj undo：撤销上一条操作
  • jj op log：查看操作历史；找回误操作很重要

和 Git 的直觉对照

  • git status -> jj st
  • git log --graph -> jj log
  • git commit --amend -> 常见是 jj desc 或直接继续改当前 @
  • git checkout <commit> -> jj edit <rev>
  • git rebase -> jj rebase
  • git reset --soft/--mixed -> 在 jj 里通常换成 edit/squash/split/rebase/restore
  • git reflog -> jj op log

个人理解

  • 先把 change 理解成“可演化的提交”
  • 先把 @ 理解成“当前正在写的那一笔”
  • jj 鼓励频繁整理历史，因为 undo 和 op log 让回退成本更低
  • 日常核心动作基本就是：st / log / desc / commit / edit / squash / split / undo
