并行编程的核心不是“多开线程”，而是管理共享状态：尽量少共享；必须共享时，明确所有权、同步边界、可见性顺序和性能成本。

  • 这不是一本并发 API 手册，而是一本讲并行程序设计方法论的书：先理解硬件，再做拆分，再选同步原语，再验证正确性，最后才谈极致性能。

核心观点：

  1. 并行编程的目标不只有性能，还有 productivity 和 generality。
  2. 并行之所以难，主要难在：
    • 工作拆分 work partitioning
    • 并发访问控制 access control
    • 资源拆分与复制 resource partitioning/replication
    • 与硬件行为对齐 hardware interaction
    • 多种能力组合后的复杂性 composite capabilities
  3. 很多问题不是“加锁就行”，而是要先决定：
    • 数据归谁管
    • 读多写少还是写多读少
    • 是否允许延迟回收
    • 是否接受近似结果
  4. 并行优化的本质往往不是“让所有线程都碰同一份共享状态”，而是减少共享、减少争用、减少全局序列化。

全书主线：

  • 前半部分讲基础：并行为什么难、硬件会怎样坑你、有哪些基本工具。
  • 中间部分讲设计：计数、分区、锁、数据所有权、延迟回收、并发数据结构。
  • 后半部分讲保障：验证、形式化验证、内存模型、未来趋势。
  • 书里最强的一条线索是：RCU 不是孤立技术，而是“读多写少 + 延迟回收 + 弱化同步成本”这一设计思想的代表。

分章节笔记：

1. How To Use This Book / Introduction

  • 作者强调：并行编程已经从科研特例，变成工程常态。
  • 三大目标彼此冲突：
    • 高性能
    • 高生产率
    • 高通用性
  • 启发：并行方案没有银弹，任何设计都在做取舍。

2. Hardware and its Habits

  • CPU、缓存、原子操作、内存屏障、cache miss、I/O 延迟，都会直接决定并行代码表现。
  • 关键认识：
    • 硬件不会按你源码的“直觉顺序”执行。
    • 缓存一致性不等于程序逻辑正确。
    • 原子操作很贵，跨核共享更贵。
  • 启发：并发 bug 和性能崩坏，很多不是算法错误，而是违背了硬件习性。

3. Tools of the Trade

  • 介绍线程、锁、读写锁、原子操作、线程局部/CPU 局部变量等。
  • 重点不在 API，而在“什么时候用哪种工具”。
  • 启发：工具选择要看争用模式，不要一上来默认 mutex。

4. Counting

  • 计数器是并行世界最容易低估的难题。
  • 书里把计数分成几类：
    • 统计型计数器：追求低开销，允许暂时不精确
    • 近似限额计数器：允许误差，但要快
    • 精确限额计数器：保证边界正确，但同步成本更高
  • 启发：先问“业务要不要绝对精确”，再决定实现。很多系统根本不需要强一致计数。

5. Partitioning and Synchronization Design

  • 并行设计第一原则是 partition first。
  • 常见策略：
    • 按数据分片
    • 按线程归属
    • 按快路径/慢路径拆分
    • 层级锁、局部缓存
  • 启发：性能通常来自“把热点拆散”，不是“把共享写得更花哨”。

6. Locking

  • 锁的问题不只是开销，还有：
    • 死锁
    • 活锁
    • 饥饿
    • 不公平
    • 可扩展性差
  • 作者对锁的态度很工程化：
    • 对应用来说，锁常常是英雄
    • 对并行库来说，锁只是工具之一
    • 对“给顺序程序强行并行化”的场景，锁可能是反派
  • 启发：锁不是过时技术，但它经常被滥用。

7. Data Ownership

  • 最强的简化手段之一：让某份数据有明确 owner。
  • 常见方法：
    • 指定线程处理
    • 函数/消息投递
    • 私有化
    • 多进程隔离
  • 启发：能用 ownership 解决的问题，通常比共享内存加锁更稳。

8. Deferred Processing

  • 这是全书最重要的章节之一。
  • 核心问题：删除/更新对象时，怎样保证并发读者还安全？
  • 书里比较了：
    • 引用计数 reference counting
    • Hazard pointers
    • Sequence locks
    • RCU
  • 大体理解：
    • 引用计数：直观，但更新路径重、扩展性一般
    • Hazard pointers：适合无锁场景，但使用复杂
    • SeqLock：读快，但读者可能重试
    • RCU：特别适合读多写少，读路径极轻
  • 启发：内存回收问题，往往才是并发设计真正难点。

9. Data Structures

  • 数据结构是否“可分区”，直接决定并行难度。
  • 书中用哈希表等例子说明：
    • 可分区结构容易扩展
    • 读多写少结构适合 RCU
    • 不可分区结构扩展难，往往需要重构设计而不是微调锁
  • 启发：性能瓶颈很多时候不是代码写法，而是数据结构选型。

10. Validation / Formal Verification

  • 并发程序不能只靠“跑过几次没出错”。
  • 书里强调：
    • tracing
    • assertions
    • static analysis
    • code review
    • benchmark / profiling
    • model checking / litmus tests / axiomatic methods
  • 一个很重要的观念：验证不是证明自己对，而是系统地找自己错。
  • 启发：并发系统的测试方法必须比普通业务代码更有攻击性。

11. Advanced Synchronization: Memory Ordering

  • 这是并发里的硬骨头。
  • 关键点：
    • 重排序来自硬件和编译器两边
    • acquire / release / barrier 不是语法细节，而是正确性边界
    • dependency、cache coherence、multicopy atomicity 都会影响结果
  • 启发：不理解内存模型，写 lock-free / RCU 类代码基本等于碰运气。

12. Putting It All Together / Advanced Topics / Future

  • 后面几章更像专题串讲：
    • 把多种同步技术混合使用
    • 非阻塞同步
    • 实时系统
    • 事务内存
    • 函数式并行
    • 对未来硬件和并行生态的判断
  • 启发：高级技术不是替代基础原则，而是建立在“分区、所有权、验证、内存序”这些基础之上。

启发点（关键洞察）：

  1. 并行设计先问“能不能少共享”，再问“怎么同步共享”。
  2. 正确的数据所有权设计，价值常常高于花哨的 lock-free。
  3. 读多写少场景，RCU 是非常强的思路。
  4. 计数器不是小问题，而是并发设计缩影。
  5. 锁的问题主要不在“慢”，而在“会形成全局串行点”。
  6. 精确性是昂贵的，很多系统应该先争取“足够正确”。
  7. 缓存、一致性、内存序会决定并发代码真实行为。
  8. 数据结构决定上限，API 技巧只能做局部优化。
  9. 并发验证要靠多种手段叠加，不能迷信单一测试。
  10. 并行编程首先是工程 discipline，不是技巧表演。

行动：

  1. 以后遇到并发性能问题，先从“是否能减少共享状态”入手，而不是直接换锁。
  2. 设计计数器时，先明确它是统计型、近似限额型，还是精确限额型。
  3. 对读多写少的数据路径，优先考虑 RCU、只读快路径和延迟回收。
  4. 对高争用热点，优先考虑分片、局部缓存、线程所有权，而不是全局原子变量。
  5. 任何涉及 lock-free、内存屏障、对象回收的代码，都要配套验证策略。

适合什么人读：

  • 内核、数据库、中间件、存储、网络系统开发者
  • 需要写高并发 C/C++/Rust/Go 基础设施的人
  • 想真正理解 RCU、内存序、可扩展同步的人

推荐阅读顺序：

  1. Introduction
  2. Hardware
  3. Tools
  4. Locking / Data Ownership
  5. Deferred Processing
  6. Validation
  7. Memory Ordering
  8. 再回头看形式化验证和高级章节

金句：

  1. 这本书真正教的不是“并发技巧”，而是把共享内存并行问题，改写成硬件能承受、团队能验证、系统能扩展的工程结构。
