编程语言

2026-04-292026-04-29
CS

S 我们用编程语言写软件，常被理解为"告诉计算机要做什么"。在 LLM 时代，这种"沟通工具"视角更被强化——既然自然语言也能让机器执行，编程语言似乎只是更啰嗦的中间层。 C 但事实是：哪怕需求被自然语言描述得再清楚，软件依然会因为边界条件、并发、错误传播而崩坏。问题不在"机器没听懂人"，而在"人自己没想清楚"。 Q 那么编程语言到底承担了什么自然语言无法承担的角色？为什么写代码这件事不能被"说清楚"替代？ A 编程语言的本质是精确建模工具，不是沟通工具。它逼你把模糊意图凝固成结构、类型、状态机和约束——而这正是软件难的地方。

三个核心论点

  1. 编程语言不是 人机沟通工具，而是 精确建模工具

    • "沟通"假设接收方有理解力，可以脑补缺失信息；"建模"要求每个细节都被明确指定，没有歧义空间。
    • 写一个 User 类型不是"告诉计算机用户长什么样"，而是把"用户"这个模糊概念固化为一组字段、不变量、生命周期。
    • 类型系统、模式匹配、所有权、effect 系统——这些机制都不是为"表达"服务，而是为"约束建模空间"服务。

  2. 软件工程的核心不是 写代码，而是把模糊业务意图转化为可执行、可维护、可验证的模型

    • 写代码只是建模的副产物。真正的工作量花在：澄清歧义、识别边界、暴露隐含假设、把"大概是这样"逼成"精确就是这样"。
    • 这也是为什么 LLM 写代码很快，但软件工程没有因此提速 10 倍——瓶颈不在打字，而在建模决策。
    • 可执行（能跑）只是底线，可维护（能改）和可验证（能信）才是建模质量的体现。

  3. 自然语言表达意图，编程语言表达约束

    • 意图是"我想要什么"，约束是"什么不能发生、什么必须成立、什么之间有关系"。
    • 软件系统真正难的地方，通常不是意图（"做一个购物车"谁都懂），而是约束（库存与订单的一致性、并发下单、退款幂等、超时回滚……）。
    • 自然语言天然擅长描述意图、不擅长描述约束——约束一多就开始矛盾、遗漏、模糊。编程语言的语法和类型系统天然为约束而生。

推论：软件工程是语言工程

从自然语言，到领域语言（DSL/Ubiquitous Language），到形式化语言（类型、契约、规范），到机器执行——每一层都是把上一层的模糊性进一步压榨为结构。

  • 业务专家用自然语言描述需求 → 充满歧义。
  • 领域建模产出领域语言 → 概念被命名、边界被划定
  • 实现层用编程语言把领域语言落到类型与函数 → 约束被机械检查。
  • 关键不变量用形式化语言（TLA+、契约、property test）兜底 → 把"我以为对"变成"机器证明对"。

所以真正在写代码时，你其实是在设计一种只为这个问题而存在的微型语言——选择什么是名词、什么是动词、什么状态合法、什么转换被禁止。代码是这种语言的实例，不是终点。

反过来理解：为什么"自然语言编程"绕不开建模

  • LLM 让"从意图到代码"的翻译变便宜了，但没有让建模本身变便宜。
  • 如果你给 LLM 的 prompt 模糊，它生成的代码就把你的模糊变成了具体——但具体得是不是你想要的，要等 bug 出现才知道。
  • 这反而强化了编程语言的价值：它是一种强迫你把意图磨成约束的介质。跳过这一步，模糊就会在运行时以事故的形式还回来。
