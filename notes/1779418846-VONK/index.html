倒排索引

  • S 场景：按关键词查海量文档（搜索引擎、ES、日志、全文索引）。
  • C 冲突：正排索引（docId → 内容）查"含某词的文档"必须 O(N) 全扫。
  • Q 问题：如何 O(1) 拿到含某词的文档列表，并支持布尔组合与相关性排序？
  • A 答案：把映射倒过来——term → posting list（含 docId、TF、位置），配合分词器、压缩与段合并。

架构

数据结构

词典（FST / B+ 树 / Hash）         Posting Lists
┌──────────┬─────┬─────────┐
│ term     │ df  │ pointer │ ──→ [(doc1, tf, [pos...]),
├──────────┼─────┼─────────┤      (doc3, tf, [pos...]),
│ "redis"  │  4  │   ●─────┼──→   (doc7, ...), ...]
│ "kafka"  │  3  │   ●     │
│ "mongo"  │  2  │   ●     │
└──────────┴─────┴─────────┘

构建

文档 → 预处理（去 HTML、归一） → 分词 → 词干化
     → 生成 (term, docId, pos, tf) → 按 term 聚合 → 写词典 + posting

查询

query → 同样的 Analyzer → terms → 查词典 → 取 posting lists
      → 求交/并集 → BM25 打分 → top-K

关键组件

  • 词典：FST（Lucene）压缩前后缀，兼顾查询与内存
  • Posting List：差值编码 + VarInt / Roaring Bitmap，加 skip list 加速求交
  • Analyzer：Character Filter → Tokenizer → Token Filter（小写/停用词/同义词/词干）
  • Segment：Lucene 风格——小 segment 不可变，后台 merge（类 LSM compaction），删除靠位图
  • 打分 BM25：
              tf · (k1 + 1)
score = idf · ─────────────────────────
              tf + k1 · (1 - b + b · |d|/avgdl)

对比

索引 查询模式 用途
倒排 term → docs 全文检索
正排 docId → 内容 取详情、过滤
B+ 树 精确/范围 关系型主键
LSM KV、范围 写密集 NoSQL
HNSW 近似最近邻 语义/向量检索

工业实现

Lucene → Elasticsearch / Solr / Tantivy；PostgreSQL GIN、MySQL FULLTEXT、ClickHouse token bloom。
