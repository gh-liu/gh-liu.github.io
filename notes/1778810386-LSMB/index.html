数据存储与检索

2026-05-212026-05-21
CS

聚焦 OLTP 场景下两类核心存储引擎家族：

  • LSM / 日志结构：写入不可变数据文件，后台再合并
  • B-Tree / 页更新：在页组织的树上就地维护有序结构

1. 问题背景

OLTP 的核心矛盾不是“能不能存”，而是：

  • 写入必须足够快，且崩溃后可恢复
  • 点查必须稳定低延迟
  • 范围查询不能太差
  • 更新和删除不能让空间无限膨胀

因此，存储引擎的本质问题是：

如何维护一个从 key -> value 或 key -> row-id 的映射， 使得写、读、空间回收、崩溃恢复之间达到可接受的平衡。

这既是 主存储 的问题，也是 二级索引 的问题。

  • 对主存储：key -> row/value
  • 对二级索引：secondary-key -> primary-key/row-id

从这个视角看，常见的 OLTP 存储结构大致分成两大家族：

  1. 日志结构存储引擎（Log-Structured / LSM 家族）
  2. 原地更新存储引擎（B-Tree 家族）

2. LSM 家族：把随机写改造成顺序写

2.1 核心思想

LSM 的基本策略是：

  • 内存中维护一个有序结构（memtable）
  • 磁盘上维护多个不可变的有序文件（SSTable / segment）
  • 新写入先顺序追加到日志，再写入内存表
  • 内存表满了再一次性刷成新文件
  • 后台持续做 merge / compaction，把多个文件合并、去重、清理墓碑

它的本质是：

写时不急着维护一个“全局已整理”的结构， 而是先便宜地记下来，读时和后台合并时再补秩序。

2.2 写路径

优点：

  • 磁盘写大多是顺序追加
  • 把随机写放大压到较低
  • 对高写入吞吐更友好

代价：

  • 一个 key 可能分散在多个文件中
  • 删除通常不是立刻物理删除，而是写入 tombstone
  • 必须依赖 compaction 回收空间与减少读放大

2.3 读路径

读取一个 key 时通常要按“新到旧”检查多个位置：

  1. 查 memtable
  2. 查最近刷盘的 SSTable
  3. 再查更老的 SSTable
  4. 遇到最新版本或墓碑后停止

为了降低这个成本，LSM 通常会配套：

  • Bloom Filter：快速判断“该文件大概率没有这个 key”
  • 分层索引：先定位 block，再进块内查找
  • block cache：缓存热点数据块

所以 LSM 的性能画像是：

  • 写友好
  • 点查可接受，但要靠过滤器和缓存
  • 范围查询能做，但跨文件 merge 让实现更复杂

2.4 Compaction 是 LSM 的代价结算器

LSM 不能只写不并。因为如果一直追加：

  • 同一个 key 会有很多旧版本
  • 读请求需要查很多文件
  • 磁盘空间会被旧版本和墓碑吃掉

所以 compaction 负责三件事：

  1. 合并多个有序文件
  2. 丢弃旧版本
  3. 清理已过期的墓碑并重写布局

这也是 LSM 最核心的后台成本来源：

  • CPU 消耗
  • I/O 消耗
  • 写放大转移到后台合并阶段

2.5 LSM 的常见变体

变体 1：按层级组织的 Leveled LSM

特点：

  • 较高层文件多、重叠多
  • 较低层按 key range 划分，层内尽量不重叠
  • 读放大较小
  • compaction 更频繁，写放大更高

适合：

  • 读延迟要求更稳
  • 不能接受查太多文件

变体 2：按批次累积的 Tiered / Size-Tiered LSM

特点：

  • 同层保留多个批次文件
  • 攒够再整体合并
  • 写放大较低
  • 读放大更高，空间放大也更明显

适合：

  • 写吞吐优先
  • 可接受更高的后台读成本

变体 3：LSM 作为二级索引

当二级索引也采用 LSM 时，本质仍是：

  • 把 secondary-key -> primary-key 记录成有序不可变文件
  • 更新时写入新版本或墓碑
  • 依靠 compaction 去重与清理

这样可以让索引写入也保持顺序化，但代价是：

  • 二级索引读取也会承受多文件查询
  • 范围查询与去重逻辑更复杂

3. B-Tree 家族：始终维护一个可查询的有序结构

3.1 核心思想

B-Tree 家族的基本策略是：

  • 磁盘被划分为固定大小的 page
  • page 组织成多叉平衡树
  • 每次写入都把新值插入或更新到树中的相应位置
  • 通过 page split / merge 维持平衡

它的本质是：

写的时候就把结构整理好， 让之后的读可以沿树稳定地走到目标位置。

相比 LSM，它更像是在持续维护一个“随时可读”的全局有序视图。

3.2 读路径

读一个 key 时：

  1. 从根 page 开始
  2. 按 key 范围向下走到内部节点
  3. 到叶子节点定位记录

特点：

  • 树高通常较低
  • 点查路径稳定
  • 范围扫描天然友好，因为叶子节点本身按 key 有序

因此 B-Tree 的一个强项是：

一旦定位到起点，后续的范围扫描可以沿叶子链表顺序读下去。

3.3 写路径

写入时通常会：

  1. 找到目标叶子页
  2. 在页内插入、更新或删除记录
  3. 页满则分裂
  4. 必要时向上递归调整父节点

优点：

  • 读路径短且稳定
  • 范围查询非常自然
  • 不需要像 LSM 那样跨多个不可变文件找最新版本

代价：

  • 磁盘层面更像随机写
  • 页分裂、页合并、碎片整理都有成本
  • 写入吞吐通常不如把随机写批量顺序化的 LSM

3.4 B-Tree 的常见变体

变体 1：B+ Tree

这是数据库里最常见的形态：

  • 内部节点只存导航键
  • 数据主要放在叶子节点
  • 叶子节点按顺序链接

这比经典 B-Tree 更适合磁盘页：

  • 扇出更大，树更矮
  • 范围扫描更高效

变体 2：聚簇与非聚簇组织

虽然不展开具体产品，但抽象上可区分两类：

  • 聚簇：叶子直接存主记录
  • 非聚簇：叶子存引用，再回到主存储取记录

这直接影响：

  • 主键范围扫描性能
  • 二级索引是否需要回表

变体 3：B-Tree 作为二级索引

这是最经典的二级索引结构：

  • secondary-key 在叶子上有序排列
  • 叶子记录指向主键或物理位置
  • 等值查找和前缀范围查找都很自然

它的优势是：

  • 二级索引查找路径稳定
  • 有序遍历和 ORDER BY / range scan 友好

它的代价是：

  • 每次写入主数据时，相关索引页也要一起维护
  • 索引越多，写放大越明显

4. 两大家族的根本取舍

4.1 对比表

维度 LSM 家族 B-Tree 家族
写入路径 先追加，再后台合并 直接更新树页
磁盘写模式 偏顺序写 偏随机写
点查 需要查 memtable + 多文件 沿树查找，路径稳定
范围查询 可做，但常需跨文件 merge 天然友好
空间回收 依赖 compaction 依赖页回收/整理
删除 tombstone + 延迟清理 直接删索引项/记录
写放大来源 compaction 页分裂、索引维护
典型优势 高写吞吐 稳定读延迟、范围查询

4.2 一个更本质的理解

两者不是“一个先进，一个落后”，而是把成本放在不同阶段：

  • LSM：写时省，读时和后台还
  • B-Tree：写时还，读时省

也可以说：

  • LSM 倾向于把“整理顺序”的工作延后
  • B-Tree 倾向于把“整理顺序”的工作当场完成

4.3 它们为什么都能同时用于主存储和二级索引

因为主存储与二级索引的共同本质都是：

维护一个按 key 可查、可更新、可恢复的有序映射。

差别只在于 value 的内容：

  • 主存储：value 是整行或记录本体
  • 二级索引：value 是主键、row-id 或指针

因此：

  • LSM 可以既做表，也做索引
  • B-Tree 可以既做表，也做索引

存储引擎真正要回答的不是“是不是索引”，而是：

  • 更新是在原位置维护，还是先追加后整理？
  • 为了更快写，愿意接受多少读放大？
  • 为了更稳读，愿意接受多少写放大？

5. 结论

如果只看 OLTP 存储引擎家族，可以把问题压缩成一句话：

LSM 通过不可变文件 + 后台合并，把随机写转换为顺序写； B-Tree 通过页式平衡树，把数据始终维持在一个可直接查询的有序结构中。

二者的差异，不在“谁支持索引”，而在：

  • 什么时候维护秩序
  • 把成本放在写时、读时，还是后台

这也是理解后续一切变体的总入口。
