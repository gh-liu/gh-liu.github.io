基于共享状态的分布式反馈控制系统

  • S 应用部署到多台机器后，需要统一管理部署、扩缩容、故障恢复和服务访问
  • C 手工运维或脚本部署难以应对机器故障、流量变化、环境差异和多服务协作
  • Q 如何让应用以声明式方式运行在集群中，并自动维持期望状态
  • A Kubernetes 通过 Pod、Deployment、Service、Controller 等抽象，把应用编排成期望状态，并持续自动调度、恢复和暴露服务

心智模型

Kubernetes 不是运行容器的工具，而是一个基于声明式 API 的分布式控制系统：用户声明期望状态，控制器持续调谐现实状态，让集群逼近目标。

  • 声明式系统（desired state）
  • 控制器模型（control loop）
  • 分布式收敛（eventual consistency）
  • 调度与资源约束

Kubernetes = API Server 中的共享状态 + 一组 Controller 的反馈调谐循环 + 一套可组合的资源抽象

你写 spec（期望状态）
    → Controller watch 实际状态
    → 发现 diff
    → 执行动作
    → 趋近期望
    → 循环（Reconcile Loop）

核心抽象

Kubernetes 不是一个大而全的应用模型，而是一组可以组合的抽象：

  • Pod 表示运行单元
  • Service 暴露访问入口
  • Config/Storage 注入运行依赖
  • Controller 维持期望状态
  • Label/Selector 把这些资源关联起来

工作负载：Pod -> Deployment(无状态) -> StatefulSet(有状态); DaemonSet(每个node一个); job/cronjob(一次性任务、定时任务) 网络：Service(应用稳定入口,L4,ClusterIP/NodePort/LoadBalancer) -> Ingress(外部流量入口,L7) 配置、存储：Config/Storage

用户浏览器 → DNS → Ingress（七层路由，按域名/路径分发）
  → Service（四层 LB，iptables 选一个 Pod IP）
  → Pod（容器处理请求）

k8s 内部

控制面组件+数据面（跑pod）

控制面: 负责维护集群大脑，API、状态存储、调度、控制循环

组件 作用 通常怎么运行
kube-apiserver Kubernetes API 入口，所有读写都先经过它(RESTful API) 通常是静态 Pod / 容器镜像
etcd 持久化保存集群共享状态，比如 Pod、Deployment、Service 等对象 通常是静态 Pod / 容器镜像，也可外部部署
kube-scheduler 给新 Pod 选择合适 Node(按资源/亲和性/污点打分选择) 通常是静态 Pod / 容器镜像
kube-controller-manager 运行各种 Controller(Deployment/ReplicaSet/Node/...)，持续调谐状态 通常是静态 Pod / 容器镜像

CoreDNS 严格来说不是 Kubernetes 控制面的核心组件，但在实际集群里非常重要，负责服务名解析，是集群服务发现的基础设施

数据面: 节点组件，每台机器上负责真正跑 Pod

组件 作用 通常怎么运行
container runtime 真正运行容器，比如 containerd、CRI-O 通常是宿主机进程，不是 Pod
kubelet 每个 Node 上的代理，负责拉起 Pod、检查容器状态、汇报 Node/Pod 状态 通常是宿主机 systemd 进程，不是 Pod
kube-proxy 实现 Service 的流量转发规则 通常是 DaemonSet Pod / 容器镜像
CNI plugin 实现 Pod 网络，比如 Calico、Cilium、Flannel 通常是 DaemonSet Pod / 容器镜像
  • apiserver 接请求
  • etcd 存状态
  • scheduler 选节点，决定放哪儿
  • controller-manager 做调谐，决定要什么
  • kubelet 跑 Pod，决定怎么跑
  • runtime 跑容器
  • CNI 管网络
  • kube-proxy 管 Service
  • CoreDNS 管服务发现

一个请求的路径：kubectl apply → apiserver → etcd 存储 → scheduler 分配 Node → kubelet 拉镜像启动容器

CNI, kube-proxy, CoreDNS

组件 输入 输出 作用
CNI Pod IP 流量 真实网络可达 Pod 网络连通
kube-proxy Service ClusterIP:Port Pod IP:Port 流量转发 / 负载均衡
CoreDNS 服务域名 Service ClusterIP 名字解析

CNI：负责 Pod 网络连通 kube-proxy / eBPF replacement：负责 Kubernetes Service 的 L4 转发 Service Mesh：负责服务间通信治理，更多是 L7 能力

Controller

给某类资源增加自动维持期望状态的能力，本质是让Pod 具备生产环境能力的自动化控制层

通常运行在 kube-controller-manager 里（少数例外），监听 API Server 中的资源变化，然后创建/修改其他资源

在 Pod 之上增加不同领域能力

  • Deployment 增加发布能力：无状态应用管理 + 滚动发布 (Deployment -> ReplicaSet -> Pod)
  • ReplicaSet 增加副本能力
  • StatefulSet 增加身份和存储能力：有状态应用管理
  • DaemonSet 增加节点级部署能力：每个节点运行一个 Pod
  • Job 增加任务完成能力
  • Node：节点生命周期管理
  • Service：服务暴露能力
  • EndpointSlice：维护 Service 后端 Pod 地址列表
  • Namespace：命名空间生命周期管理
  • Garbage Collector：资源级联删除
  • ServiceAccount：自动创建和管理身份

Scheduler

Scheduler 负责给还没有分配 Node 的 Pod选择一个合适的 Node，并把绑定结果写回 apiserver。

  1. 过滤：哪些 Node 能跑？
  2. 打分：哪个 Node 更合适？

Scheduler vs. Controller:

Controller 造 Pod，Scheduler 选 Node，kubelet 跑容器。

组件 负责什么
Controller 让资源数量、版本、状态逼近期望
Scheduler 给未调度的 Pod 选择 Node
kubelet 在目标 Node 上真正启动 Pod

Operator

内置 Controller 覆盖了通用场景，但如果业务有特殊的运维逻辑（如数据库主从切换、证书自动续期），如何让 k8s 理解并自动管理自定义的业务资源？

CRD（自定义资源）+ Controller（自定义控制循环）= Operator

本质还是声明式 + 控制循环，只是你自己定义了"期望状态"长什么样、"如何趋近"的逻辑。
