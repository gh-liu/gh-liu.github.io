  • Q 如何衡量、量化并提升系统的可用性?有哪些常用的工程模式?
  • A 通过 故障转移 (Fail-over) 与 复制 (Replication) 两种互补模式实现高可用,并用"几个 9"来量化;在拓扑结构上尽量将关键组件 并联 而非串联。

一、核心思路

高可用 (HA) 由两类互补模式支撑:

  1. Fail-over —— 主机失效时由备机接管。
  2. Replication —— 多副本同时承载/同步数据,既提升可用性又降低数据丢失风险。

二、Fail-over 故障转移

2.1 Active-Passive(主备 / master-slave)

  • 主、备服务器之间通过 heartbeat 心跳检测。
  • 心跳中断 → 备机接管主机 IP,继续提供服务。
  • 停机时长取决于备机状态:
    • Hot standby:备机已运行,接管时间短。
    • Cold standby:需冷启动,接管时间长。
  • 仅主服务器处理流量。

2.2 Active-Active(双主 / master-master)

  • 两台服务器同时处理流量,负载均摊。
  • 路由方式:
    • 对外:DNS 须包含两台服务器的公网 IP。
    • 对内:应用层须感知两台服务器。

2.3 Fail-over 的缺点

  • 引入额外硬件与复杂度。
  • 主机故障时,尚未复制到备机的新写入数据可能丢失。

三、Replication 复制

  • Master-slave replication:写主、读从,适合读多写少。
  • Master-master replication:多写入点,需处理写冲突。
  • 详见 Database 章节。

四、可用性的量化(几个 9)

可用性常以服务正常运行时间百分比表示,以"9"的个数衡量。

等级 年停机 月停机 周停机 日停机
99.9% (3 个 9) 8h 45m 57s 43m 49.7s 10m 4.8s 1m 26.4s
99.99% (4 个 9) 52m 35.7s 4m 23s 1m 5s 8.6s

五、串联 vs 并联可用性

服务由多组件组成时,整体可用性依赖于组件的拓扑关系。

5.1 串联(In Sequence)

任意一个组件故障即导致整体故障,整体可用性下降:

Availability(Total) = Availability(Foo) × Availability(Bar)

例:99.9% × 99.9% = 99.8%

5.2 并联(In Parallel)

只要任一组件可用即整体可用,整体可用性提升:

Availability(Total) = 1 − (1 − Availability(Foo)) × (1 − Availability(Bar))

例:两个 99.9% 组件并联 → 99.9999%

六、关键洞察 / Takeaways

  • 冗余(并联)是提升可用性的核心手段;依赖链越长(串联),整体可用性越差。
  • 设计高可用系统时:
    • 尽量将关键组件 并联部署(多副本 / 多 AZ / 多区域)。
    • 谨慎评估并缩短强依赖链路。
    • 在 Fail-over 之外配合 Replication,降低主机故障导致的数据丢失风险。
  • 选择 Active-Passive 还是 Active-Active,要在 复杂度、成本、停机容忍度、写冲突 之间权衡。
