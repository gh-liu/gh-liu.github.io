Go: panic

panic 是 Go 里的程序无法继续按当前控制流执行的信号。

通常和 recover 一起理解：panic 负责触发栈展开，recover 负责在 defer 中截住这次展开。

它不是常规错误处理机制；一旦触发，当前函数会停止正常执行，runtime 开始沿调用栈展开，并依次执行已经注册的 defer。

panic 是什么

  • panic 接收任意值 v
  • 可以由代码显式触发：panic(...)
  • 也可以由 runtime 触发，比如数组越界、nil 指针解引用
  • 如果 panic 一路展开到 goroutine 栈顶仍未被 recover，进程会打印堆栈并退出

何时会 panic

两类：

  1. 显式 panic(...)
  2. runtime panic

常见 runtime panic：

  • 数组 / slice 越界
  • nil 指针解引用
  • 向已关闭的 channel 发送
  • 并发读写普通 map
  • 类型断言失败：x := i.(T)

panic 发生后会怎样

核心流程：

  1. 当前函数停止正常执行
  2. 当前函数注册的 defer 按 LIFO 顺序执行
  3. 当前函数返回到调用者，但不是正常返回，而是继续带着 panic 向上展开
  4. 调用栈上的每一层都重复这个过程
  5. 若某层 defer 中成功 recover，展开停止，控制流从该函数返回
  6. 若直到 goroutine 栈顶都没人恢复，程序崩溃

panic 和 defer

panic 最重要的配套机制是 defer。

  • defer 保证清理逻辑仍然执行
  • 所以即使程序正在异常展开，也可以关闭文件、解锁 mutex、记录日志

how to read panic

panic: <panic 值>

goroutine N [状态]:
<包名>.<函数名>(<参数>)
    <文件路径>:<行号> +0x<偏移>
...

+0x<偏移> 是相对这个函数起始位置的偏移

阅读顺序：

  1. 先看第一行 panic: ...
  2. 再看最上面的业务栈帧：通常就是直接出错点
  3. 往下看是谁调用了它
  4. 若是 runtime panic，优先看第一个非 runtime / 标准库的业务文件行号

panic 和 error 的边界

经验上：

  • error：预期内失败，属于业务控制流的一部分
  • panic：违反不变量、程序员错误、系统已进入无法继续信任的状态

适合返回 error：

  • 文件不存在
  • 网络超时
  • 参数非法但属于外部输入
  • 下游服务失败

适合 panic：

  • 明确不应该发生的分支
  • 初始化阶段发现关键依赖缺失，程序无法启动
  • 内部状态被破坏，不再满足不变量

MustXxx 风格本质上是：把失败升级为程序不应继续。

runtime 视角

从机制上看，panic 不是 Java 那种语言级异常体系；它更像 runtime 驱动的栈展开流程：

  • 每个 goroutine 维护自己的调用栈
  • panic 只影响当前 goroutine
  • 展开时 runtime 会逐层执行该 goroutine 上的 defer
  • 若无人恢复，runtime 最终打印当前 panic 的信息和堆栈

所以：

  • panic 不是跨 goroutine 传播的
  • 其他 goroutine 不能替你恢复当前 goroutine 的 panic

最佳实践

  1. 不要用 panic 代替常规 error 返回
  2. 可以在程序入口、worker 边界、HTTP middleware 边界统一 recover
  3. 在 defer 里做资源清理，而不是依赖正常返回
  4. 库代码一般不要随意 panic，除非是明确的 programmer error
  5. 看到 panic，先定位第一个业务代码栈帧，不要先钻 runtime 栈
