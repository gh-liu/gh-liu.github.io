分布式数据系统

2026-06-122026-06-12
CS

理解一个分布式数据系统，就是理解它如何在不可靠的分布式环境中，把某类数据问题抽象成稳定的读写语义，并通过数据面、控制面和运维面持续维持这些语义。

心智模型

  1. 问题：它解决什么类型的数据问题？
    • 业务需要什么能力？缓存、队列、搜索、事务、分析、时序、对象存储？
    • 主要访问模式是什么？读多、写多、点查、范围查、聚合、流式消费？
  2. 语义：它向使用者承诺什么？
    • 数据模型是什么？KV、文档、关系、列式、日志、图？
    • 读写语义是什么？一致性、事务、隔离、顺序、可见性、失败后的结果？
  3. 机制：它如何让语义成立？
    • 数据面：一次读写请求如何完成。
    • 控制面：系统如何长期维持拓扑、复制、恢复、均衡和演进。
    • 运维面：系统如何被观察、诊断、保护和恢复。
  4. 边界：它在什么条件下工作得好，什么条件下开始失效？
    • 优化了什么，所以适合什么。
    • 牺牲了什么，所以不适合什么。
    • 成本、复杂度、故障模式和规模上限在哪里。

系统拆解清单

系统拆解清单是对心智模型的展开：系统定位回答问题和语义，基础架构回答机制，适用场景、权衡边界和运维侧回答边界。

  1. 系统定位
    • 它解决什么问题？
    • 它抽象出了什么模型？
    • 它向用户承诺什么语义？
  2. 基础架构
    • 数据面：读写路径如何运行？
    • 控制面：如何让读写路径长期正确运行？
    • 核心机制：分片、复制、一致性、恢复、调度如何支撑语义？
  3. 适用场景
    • 业务场景决定访问模式。
    • 访问模式决定数据模型、索引方式、分区方式和存储结构。
    • SLO 决定延迟、吞吐、可用性、一致性和成本要求。
  4. 权衡边界
    • 优化了什么，所以适用于什么？
    • 牺牲了什么，所以不适用于什么？
    • 规模、成本、复杂度和故障模式的边界在哪里？
  5. 运维侧
    • 如何长期稳定运行、观察、诊断和恢复？
    • 持续验证什么：读写路径是否健康，控制面是否稳定，资源是否足够，数据是否安全。
    • 监控什么：节点健康、读写路径、存储路径、控制面状态、客户端体验。
    • 如何定位问题：现象 -> 影响面 -> 路径 -> 瓶颈 -> 控制面状态 -> 最近变更 -> 恢复动作。
